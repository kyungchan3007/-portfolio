🤖 원격 제어 및 모니터링

2025.08 – 2025.12 · ㈜TSM Technology · 과장 · FE 개발 · AWS 서버 구축

AWS IoT Core 기반 현장 장비 원격 제어 및 실시간 모니터링 시스템.
MQTT direct write 구조를 Lambda 중간 처리 계층으로 전환해, 제어 응답 지연과 중복 저장 문제를 함께 줄였습니다.
CloudWatch Alarm 연동으로 장애 대응 체계 확보.

문제를 어떻게 풀었는가​

원격 제어 시스템에서는 명령을 보내는 것보다, 그 명령이 장비에 반영되고 다시 화면에 안정적으로 동기화되는 전체 흐름이 더 중요했습니다.

문제 실시간 제어 시스템 특성상 응답 지연과 메시지 수신 불안정은 곧바로 사용자 불신으로 이어질 수 있었습니다. 실제로 제어 명령 이후 같은 시각의 동일 데이터가 여러 건씩 중복 저장되고 있었고, 이 중복 적재가 DB 부담과 체감 지연을 함께 키우고 있었습니다.

발견 원인을 추적해 보니 병목은 개별 화면이나 단일 함수가 아니라, Socket.IO · MQTT · DynamoDB로 이어지는 흐름 전반에 분산되어 있었습니다. 그중 가장 큰 문제는 초기 구조에서 MQTT로 들어온 메시지가 DynamoDB에 직접 적재되고 있었다는 점이었습니다. 같은 초 단위의 동일 데이터가 여러 건씩 쌓이면서 저장 부담과 응답 지연이 함께 커지고 있었습니다.

  • MQTT 메시지가 직접 저장되는 구조라 중복 데이터 적재를 제어할 중간 처리 계층이 없었습니다.
  • 화면마다 소켓 연결을 따로 만들면서 연결 준비와 갱신 흐름이 분산되어 있었습니다.
  • 리스너 정리 방식이 거칠어 필요한 이벤트까지 함께 제거되거나 중복 등록될 위험이 있었습니다.
  • MQTT 메시지를 클라이언트 연결 시마다 다시 수신하도록 구성해 불필요한 중복 처리와 전파 비용이 발생했습니다.
  • DB 저장 리스너가 반복 루프 안에서 재등록되며 수신 안정성이 크게 떨어지고 있었습니다.

개선 그래서 저는 단순히 일부 코드를 최적화하는 대신, 실시간 제어 흐름 자체를 다시 정리하는 방식으로 접근했습니다.

  • 가장 먼저 AWS Lambda를 중간 처리 계층으로 추가해 중복 검사와 저장 제어를 먼저 수행하도록 바꿨습니다.
  • 소켓 연결을 공통 모듈로 통합해 화면 간 연결 재사용 구조로 바꿨습니다.
  • 화면이 등록한 리스너만 정확히 정리하도록 cleanup 기준을 수정했습니다.
  • MQTT는 서버가 한 번만 수신하고, 클라이언트에는 단일 브로드캐스트 구조로 전파하도록 바꿨습니다.
  • DB 저장 리스너는 주기 루프 밖에서 한 번만 등록하도록 옮겨 수신 안정성을 확보했습니다.

이유 실시간 제어 시스템에서는 한 지점만 빠르게 만드는 것보다, 명령 전달부터 상태 반영까지 이어지는 전체 흐름에서 중복 저장과 불필요한 재연결을 제거하는 것이 더 중요했기 때문입니다. 특히 direct write 구조를 끊지 않으면 저장 부담과 지연이 계속 누적되기 때문에, 먼저 중간 제어 계층을 세워 흐름을 통제할 필요가 있었습니다.

결과 그 결과 제어 응답 지연을 10초 이상에서 1초 이내로 줄였고, 중복 저장을 막아 DB 부담과 이후 조회 부담도 함께 낮출 수 있었습니다. 그 위에 소켓 생성 지점과 중복 리스너를 정리하면서 수신 안정성과 화면 반영 속도도 추가로 개선할 수 있었습니다. 아래는 그 과정에서 실제로 손본 핵심 구조들입니다.

  • 실시간 제어 특성상 일반 조회와 같은 구조로는 흐름을 안정적으로 통제하기 어려워, 실시간 조회·일반 조회·socket 설정을 분리할 수 있는 VSA 구조를 적용했습니다.
  • 이 선택 덕분에 실시간 처리 경계와 화면 책임을 분리한 상태에서 제어 흐름 개선을 안정적으로 누적할 수 있었습니다.
  • 또한 중복 메시지 처리와 멱등성 대응을 제어 흐름 안에 포함시켜, 저장 이전 단계에서 불필요한 중복을 먼저 걸러낼 수 있게 만들었습니다.

1. 소켓 생성 지점 통합​

화면마다 직접 Socket.IO 연결을 생성하던 구조에서는 화면 전환이나 재진입 시 연결 준비와 상태 동기화 흐름이 쉽게 흔들렸습니다.

그래서 소켓 연결을 공통 소켓 모듈로 통합해, 제어와 모니터링이 같은 연결 기준 위에서 동작하도록 만들었습니다.

예시 코드입니다. 일반적인 패턴을 기반으로, 도메인 특성에 맞게 재구성해 적용했습니다.

shared-socket-pattern.ts
let socketInstance: Socket | null = null;

export function getSharedSocket() {
  if (!socketInstance) {
    socketInstance = connectSocket();
  }

  return socketInstance;
}

2. 메시지 리스너 cleanup 적용​

전체 리스너를 강제 제거하던 방식에서는 다른 화면에서 필요한 이벤트까지 함께 지워질 수 있어, 갱신 누락과 재등록 문제가 함께 발생할 수 있었습니다.

그래서 화면에서 등록한 리스너만 정확히 제거하도록 변경해, 실시간 갱신 흐름이 서로 간섭하지 않게 만들었습니다.

예시 코드입니다. 일반적인 패턴을 기반으로, 도메인 특성에 맞게 재구성해 적용했습니다.

listener-cleanup-pattern.ts
useEffect(() => {
  const socket = getSharedSocket();

  const handleMessage = (payload: unknown) => {
    updateView(payload);
  };

  socket.on('message', handleMessage);

  return () => {
    socket.off('message', handleMessage);
  };
}, []);

실제로는 화면에 필요한 비즈니스 조건을 먼저 확인한 뒤, 해당 결과만 화면 상태에 반영했습니다.

3. MQTT 브로드캐스트 구조 개선​

클라이언트 연결 시마다 MQTT 리스너를 재등록하던 구조에서는 연결 수가 늘어날수록 불필요한 중복 처리와 전파 비용이 함께 커졌습니다.

그래서 서버에서 한 번만 수신하고 전체 클라이언트에 브로드캐스트하는 구조로 바꿔, 메시지 흐름을 단순화했습니다.

예시 코드입니다. 일반적인 패턴을 기반으로, 도메인 특성에 맞게 재구성해 적용했습니다.

broadcast-pattern.ts
export function bindBroadcast(io: Server, source: EventSource) {
  source.on('message', (payload) => {
    io.emit('message', payload);
  });
}

4. DB 저장 리스너 위치 개선​

60초 주기 루프 내부에서 MQTT 리스너를 재등록하던 구조는 수신 안정성을 떨어뜨리고, 반복 실행 과정에서 메시지 누락 가능성까지 만들고 있었습니다.

그래서 리스너를 루프 밖에서 한 번만 등록하도록 변경해, 상태 저장 흐름이 주기 작업과 분리된 안정적인 구조로 동작하게 만들었습니다.

예시 코드입니다. 일반적인 패턴을 기반으로, 도메인 특성에 맞게 재구성해 적용했습니다.

persist-pattern.ts
function bindPersistence(source: EventSource) {
  source.on('message', async (payload) => {
    await persist(payload);
  });
}

실제로는 메시지를 필요한 형태로 해석한 뒤 저장 로직을 실행했고, 저장 결과와 화면 반영 흐름은 분리해서 관리했습니다.
