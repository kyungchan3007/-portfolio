⚡ BEMS

신규 개발 2023.08 – 2025.08 · 운영·리팩터링 2025.09 – 현재 · ㈜TSM Technology · 과장 · FE 개발 · 팀 리딩

운영 지표 데이터 실시간 수집·시각화 및 계층형 데이터 조회·집계 분석 시스템.
이 프로젝트에서는 단순히 기능을 구현하는 수준을 넘어서, 상태 관리, 캐시, 실시간 처리, 렌더링 성능, 크로스 브라우징, 품질 검증까지 프론트엔드 전반의 기준을 만들고 운영 가능한 수준으로 끌어올렸습니다.

기술 스택​

React TypeScript Zustand TanStack Query BFF Web Worker DevExtreme Vitest Playwright Storybook SonarQube

성과 요약​

항목문제적용결과
상태·캐시 구조동일 데이터를 여러 페이지에서 반복 요청Zustand·TanStack Query·BFF로 상태 소유권과 캐시 경계 분리브라우저 API 요청 40회 → 10회, 75% 감소
실시간 처리1분 단위 데이터 전환 시 네트워크와 메인 스레드 병목 발생Delta Update + Web Worker 분리네트워크 전송량 약 60% 감소, 화면 반영 3~5초 → 1초 이내
크로스 브라우징Worker 반영 이후 Grid·Chart 잘림, 이미지 미노출, 이중 스크롤 발생requestAnimationFrame, ResizeObserver, 레이아웃 보정Chrome·Edge·Safari·Firefox 동작 일관성 확보
품질 검증동일 기준 UAT 품질 이슈가 반복적으로 발생SonarQube, AI Reviewer, Vitest, Playwright, Storybook, UAT 연결UAT 품질 이슈 500건 → 10건, 약 98% 감소
협업 기준요구사항·검증 기준·변경 이력이 맞지 않아 재작업 발생요구사항 문서화, 검증 결과 기록, 변경 이력 관리정기 협의 회의 주 3회 → 1회
인증 보안외부 SP가 SAML 요청-응답 상관관계(UUID)를 검증하지 않아 인증 위조 가능성 존재FE-BE 간 별도 state 토큰을 httpOnly 쿠키로 발급·대조하는 검증 계층 추가요청-응답 상관관계 검증 확보, 약 3주 내 대응 완료

이 프로젝트에서 맡은 역할​

  • 서비스 설계, 프론트엔드 구현, 클라이언트 커뮤니케이션, 일정 조율을 주도했습니다.
  • 상태 관리, 캐시, 실시간 처리, 품질 검증을 각각 따로 최적화하지 않고 하나의 운영 구조로 연결했습니다.
  • 기능을 만드는 일뿐 아니라, 요구사항과 검증 기준을 문서화해 팀이 같은 기준으로 논의할 수 있게 만들었습니다.

핵심 문제​

BEMS의 핵심 과제는 기능 추가보다도, 늘어나는 데이터와 요구사항 속에서 실시간 서비스의 성능과 정합성을 계속 유지하는 일이었습니다.

실제로 병목은 다음처럼 여러 층에서 동시에 드러났습니다.

  • 서버 데이터와 UI 상태가 혼재돼 불필요한 리렌더링이 발생했습니다.
  • 동일 API를 여러 페이지에서 반복 호출했습니다.
  • 1분 단위 데이터 전환 이후 비교·갱신 연산이 메인 스레드를 점유했습니다.
  • Worker 결과가 비동기로 반영되면서 Grid·Chart·이미지·스크롤이 브라우저마다 다르게 깨졌습니다.
  • 기능 변경이 반복되는데 요구사항, 검증 결과, 변경 이력이 일치하지 않아 품질 이슈와 재작업이 누적됐습니다.

이 프로젝트는 프론트엔드에서 무엇을 만들 수 있는지보다, 운영 서비스를 어떻게 더 빠르고 안정적이며 검증 가능한 구조로 바꿀 수 있는지를 보여주는 프로젝트였습니다.

1. Zustand · TanStack Query · BFF 상태·캐시 구조​

페이지가 늘어날수록 서버 데이터와 UI 상태가 한 덩어리로 섞이면 상태 변경 범위도 불명확해지고, 동일 데이터를 여러 화면에서 다시 불러오게 됩니다.

그래서 BEMS에서는 상태 소유권을 분리했습니다.

  • Zustand는 선택 상태, 탭, 필터 등 클라이언트 UI 상태 담당
  • TanStack Query는 서버 데이터 조회·캐싱·리페치 담당
  • 공통 Query Key로 페이지 간 동일 데이터 재사용
  • BFF는 인증과 사용자별 서버 캐시 담당

결과​

  • 동일 사용자 플로우의 브라우저 API 요청 40회 → 10회
  • 중복 브라우저 요청 75% 감소
  • 새로고침·토큰 갱신 시 BFF 캐시로 Origin 재호출 방지
domain.ts
// 설명용 예시: 실제 Store·Query 이름이 아님
const 도메인스토어 = create<{
  선택값: string | null
  선택값설정: (id: string) => void
}>((set) => ({
  선택값: null,
  선택값설정: (id) => set({ 선택값: id }),
}))

export const 도메인키 = {
  detail: (id: string) => ['domain', 'detail', id] as const,
}
domain.ts
// 설명용 예시: 실제 캐시 구현·Key가 아님
export async function 데이터조회(사용자ID, 대상ID) {
  const 캐시키 = "사용자·대상 기준 캐시 키 생성"
  const 캐시값 = await cache.get(캐시키)

  if (캐시값) "캐시된 값 즉시 반환"

  const 원본데이터 = await requestOrigin(대상ID)
  await cache.set(캐시키, 원본데이터, "TTL 설정")

  return 원본데이터
}

2. Delta Update · Web Worker 기반 실시간 처리​

고객 요구사항 변경으로 기존 15분 주기 조회를 1분 단위 구조로 바꾸면서, 단순히 조회 주기를 줄이는 방식으로는 병목을 감당할 수 없었습니다.

문제​

  • 전체 데이터를 반복 수신하면 전송량이 빠르게 늘어남
  • 비교·갱신 연산이 메인 스레드를 점유함
  • 4,400개 데이터 비교·캐싱 시 UI 블로킹이 발생함

적용​

  • 이전 결과를 캐싱하고 변경 데이터만 갱신하는 Delta Update 적용
  • 시간·컬럼 단위 비교로 시계열 오염 방지
  • 비교·검증 연산을 Web Worker로 분리
  • performance.now()와 requestAnimationFrame()으로 수신 시점과 반영 시점 측정

결과​

  • 네트워크 전송량 약 60% 감소
  • 화면 반영 3~5초 → 1초 이내
  • 메인 스레드 점유 최소화
domain.ts
// 설명용 예시: 실제 Worker 메시지와 데이터 구조가 아님
self.onmessage = ({ data }) => {
  const 변경목록 = data.next.filter((항목) => "이전 값과 비교해 변경된 항목만 추출")

  self.postMessage({ changed: 변경목록 })
}
domain.ts
// 설명용 예시: 실제 React 상태 갱신 코드가 아님
worker.onmessage = ({ data }) => {
  setRows((현재값) => 행병합(현재값, data.changed))

  requestAnimationFrame(() => {
    grid.updateDimensions()
  })
}

3. Cross Browsing 렌더링 최적화​

Worker 결과가 비동기로 반영되고, 탭 전환과 레이아웃 변경이 겹치면 브라우저별로 렌더링 문제가 발생했습니다.

대표적인 문제는 아래와 같았습니다.

  • Grid·Chart 잘림
  • 이미지 미노출
  • 이중 스크롤
  • 브라우저 Resize 이후에만 정상 표시

그래서 DOM 반영 이후 requestAnimationFrame으로 후속 처리를 예약하고, ResizeObserver로 컨테이너 변경을 감지해 Grid·Chart 크기를 다시 계산하도록 정리했습니다. 또한 flex 자식의 min-width: 0, min-height: 0를 명시해 레이아웃 계산이 브라우저마다 달라지는 문제를 줄였습니다.

domain.ts
// 설명용 예시: 실제 Grid API와 변수명이 아님
useEffect(() => {
  const 대상요소 = containerRef.current
  if (!대상요소) return

  const observer = new ResizeObserver(() => {
    requestAnimationFrame(() => grid.updateDimensions())
  })

  observer.observe(대상요소)
  return () => observer.disconnect()
}, [])
domain.css
/* 설명용 예시: 실제 Class 이름이 아님 */
.domain-layout__content {
  flex: 1;
  min-width: 0;
  min-height: 0;
  overflow: auto;
}

이 대응으로 Chrome, Edge, Safari, Firefox에서 Grid·Chart·이미지·스크롤 동작의 일관성을 확보했습니다.

4. 품질 검증 체계 정리​

실시간 성능을 개선해도 품질 이슈가 반복되면 운영 프로젝트로서는 불완전합니다. 그래서 BEMS에서는 검증 흐름 자체를 구조화했습니다.

  • SonarQube: 정적 분석과 코드 컨벤션 점검
  • AI Reviewer: 비즈니스 로직, 인증/보안, 캐시 정책 사전 점검
  • Vitest: 함수와 상태 로직 검증
  • Playwright: 사용자 흐름 검증
  • Storybook: UI 상태와 회귀 확인
  • UAT: 동일 기준 최종 확인

결과​

  • 동일 기준 UAT 품질 이슈 500건 → 10건
  • UAT 품질 이슈 약 98% 감소

이 흐름은 “문제가 생기면 나중에 잡는 방식”이 아니라, 변경이 들어올 때 어떤 계층에서 어떤 문제를 먼저 막을지 기준을 세운 과정이었습니다.

5. 협업 기준과 문서화​

기능이 많고 변경 요청이 잦을수록, 요구사항과 검증 기준이 어긋나는 순간 성능 문제보다 커뮤니케이션 비용이 더 커집니다.

그래서 아래 기준을 함께 정리했습니다.

  • 요구사항 문서화
  • 검증 결과 기록
  • 변경 이력 관리
  • 개발 컨벤션 정리

결과​

  • 정기 협의 회의 주 3회 → 1회
  • 반복 확인과 오해로 인한 재작업 감소

6. SAML SSO 요청-응답 검증 계층 구축​

외부 제어 시스템은 SP-Initiated SAML 방식으로 로그인을 처리합니다. 정상적인 흐름이라면 SP(서비스 제공자)가 로그인 요청 시 발급한 고유 UUID를, IdP(Identity Provider)가 인증을 마치고 돌려주는 응답에서 다시 검증해 "내가 실제로 보낸 요청에 대한 응답이 맞는지" 확인해야 합니다.

문제​

외부 제어 시스템과 BEMS 간 SSO 연동 과정에서 이 UUID 검증 단계가 비어있는 것을 코드 분석으로 확인했습니다.

  • SP가 응답에 담긴 UUID를 검증하지 않아, 요청과 응답의 상관관계를 확인할 방법이 없었습니다.
  • 현지 개발팀은 UUID 확인 로직이 없으므로 애초에 검증하지 않는 구조라고 판단했습니다.
  • UUID 검증이 없다면 이론상 정당한 요청 없이도 응답을 위조해 접근할 수 있어, 정상 인증 흐름에서 반드시 있어야 할 단계가 빠진 상태였습니다.

적용​

외부 시스템의 SP 코드를 수정할 수 없는 상황이라, 애플리케이션 레벨에서 별도의 요청-응답 상관관계 검증 계층을 추가했습니다.

  • 로그인 요청 시점에 프론트에서 먼저 해당 계정이 BEMS 사용자인지 백엔드에 확인
  • random로 생성한 state 값을 httpOnly 쿠키에 임시 저장 — 이 값은 SAML 왕복과는 무관하게 프론트-백엔드끼리만 주고받는 별도 채널로 운용
  • 백엔드가 IdP로부터 SAML 인증 결과를 받으면, 그 결과를 프론트로 전달
  • 프론트는 인증 성공 여부와, 자신이 쿠키에 보관 중인 state 값이 백엔드가 돌려준 값과 일치하는지 문자열로 비교해 최종 로그인 여부 결정

OAuth의 state 파라미터가 CSRF를 막는 것과 같은 원리로, SP가 하지 않는 요청-응답 바인딩을 프론트-백엔드가 자체적으로 만든 셈입니다.

결과​

  • 외부 SP가 검증하지 않는 요청-응답 상관관계를 애플리케이션 레벨에서 확보
  • httpOnly 쿠키 사용으로 state 값의 XSS 탈취 위험 차단
  • 코드 분석부터 검증 계층 적용까지 약 3주 내 대응 완료
domain.ts
// 설명용 예시: 실제 API·쿠키 이름이 아님

// 1. 사용자 확인 — 등록된 사용자인지 확인
export async function 로그인요청(사용자ID) {
  const 등록여부 = await 사용자확인(사용자ID)

  if (!등록여부) "미등록 사용자 오류로 종료"

  // 2. state 값 생성 후 httpOnly 쿠키에 저장
  "randomBytes 기반 state 값 생성"
  "httpOnly 쿠키에 저장"

  "IdP로 리다이렉트"
}

// 3. IdP 응답 이후 콜백에서 검증
export function 콜백검증(인증성공, 응답값) {
  const 저장값 = "쿠키에서 state 값 조회"
  return 인증성공 && "저장값과 응답값 일치 여부 비교"
}

이 대응 이후, 외부 시스템의 SAML 인증 결과와 무관하게 BEMS 자체적으로 요청-응답 상관관계를 검증할 수 있는 최소한의 방어선을 확보했습니다.

BEMS는 단순히 성능을 올린 프로젝트가 아니라, 기능·품질·협업 기준을 같이 정리해 운영 가능한 프론트엔드 구조를 만든 프로젝트였습니다.
