♿ 웹 접근성 개선

FMS · BEMS · 원격 제어 시스템 공통 적용 · ㈜TSM Technology · FE 개발

보이는 UI를 쓸 수 있는 UI로 끌어올린 작업입니다.
운영 중인 서비스는 연령대가 높은 사용자와 현장 운영자가 함께 쓰기 때문에, 접근성 문제가 곧 조작 실수와 업무 지연으로 이어집니다.
저는 heading·landmark·ARIA·포커스·색 대비를 전면 정비하고, axe + Playwright 기반 자동 점검으로 회귀를 막는 체계를 세 개 서비스에 같은 기준으로 적용했습니다.

기술 스택​

Next.js App Router React TypeScript Semantic HTML WAI-ARIA focus-visible Playwright @axe-core/playwright

성과 요약​

항목문제적용결과
문서 구조heading 순서가 어긋나고 main landmark가 중복heading 계층 정리, 중복 main 제거 및 landmark 재정비스크린리더 탐색 경로 정상화
탭 UI 관계탭과 탭패널이 시각적으로만 연결되어 보조기기에서 관계 파악 불가role·aria-controls·aria-labelledby로 관계 명시탭 전환 시 현재 위치와 대상 패널 전달
진행 상태 전달progressbar에 접근 가능한 이름이 없어 "진행률 몇 %"만 읽힘접근 가능한 이름과 값 속성 부여무엇의 진행 상태인지 음성으로 전달
키보드 조작링크·버튼 포커스 위치가 화면에서 보이지 않음focus-visible 기반 포커스 링 스타일 추가마우스 없이도 현재 위치 식별 가능
가독성요약 카드·푸터·내비게이션의 저대비 텍스트텍스트 색상 대비 개선저시력·고령 사용자 판독성 향상
회귀 방지화면이 추가될 때마다 접근성 품질이 다시 떨어짐@axe-core/playwright E2E 자동 점검 도입공개·인증 주요 8개 경로 점검 통과

이 작업에서 맡은 역할​

  • 접근성을 권장사항이 아니라 릴리즈 전 확인하는 품질 기준으로 정의하고 팀에 공유했습니다.
  • 세 개 서비스의 heading·landmark·ARIA 관계·포커스·색 대비를 직접 점검하고 정비했습니다.
  • 수동 점검에 의존하지 않도록 axe 기반 자동 점검을 E2E 파이프라인에 연결했습니다.

핵심 문제​

세 서비스 모두 화면상으로는 정상 동작했습니다. 문제는 시각적으로 동작한다는 것이 사용 가능하다는 뜻은 아니었다는 점입니다.

  • 스크린리더로 읽으면 heading 순서가 어긋나 문서 구조를 파악할 수 없었습니다.
  • 모바일/반응형 중심으로 UI를 다듬는 과정에서 landmark, 대비, 포커스 상태가 쉽게 누락됐습니다.
  • 탭·프로그레스바처럼 상태를 가진 UI가 보조기기에는 아무 의미도 전달하지 못했습니다.
  • 한 번 고쳐도 다음 화면이 추가되면 같은 문제가 다시 생겼습니다.

FMS·BEMS·원격 제어처럼 정보 전달의 명확성과 조작 안정성이 중요한 시스템에서는, 이런 문제가 단순 불편이 아니라 오조작과 업무 지연으로 이어집니다. 특히 사용자 연령대가 높거나 현장에서 빠르게 정보를 식별해야 하는 환경일수록 영향이 큽니다.

그래서 목표를 두 가지로 잡았습니다. 지금 있는 화면을 실제로 쓸 수 있게 만들 것, 그리고 이후 기능이 추가돼도 그 기준이 유지되게 만들 것.

1. 문서 구조 정리 — landmark와 heading부터​

문제. heading이 시각적 크기 기준으로 선택되어 h1 다음에 h4가 오는 식으로 계층이 어긋나 있었고, 레이아웃 리팩토링 과정에서 main landmark가 중복 선언된 화면이 있었습니다. 스크린리더 사용자는 heading 목록으로 문서를 훑기 때문에, 이 상태에서는 페이지 구조 자체를 파악할 수 없습니다.

적용.

  • heading을 의미 계층 기준으로 재배치하고, 크기는 스타일로만 조정
  • 중복 main 제거 및 header·nav·main·footer landmark 정리
  • 본문으로 바로 이동하는 skip link 추가
// 설명용 예시: 실제 레이아웃 구조가 아님
<a href="#main-content" className="skip-link">
  본문으로 바로가기
</a>

<main id="main-content">
  {children}
</main>

결과. 키보드 사용자는 반복되는 내비게이션을 건너뛰고 본문으로 바로 진입할 수 있고, 스크린리더 사용자는 heading 목록만으로 화면 구조를 파악할 수 있게 됐습니다.

2. 탭·진행 상태 UI의 ARIA 관계 설계​

문제. 탭과 탭패널이 스타일로만 연결되어 있어, 보조기기에서는 "어떤 탭이 선택됐고 그 탭이 어떤 영역을 제어하는지"가 전달되지 않았습니다. progressbar도 마찬가지로 값만 읽히고 무엇의 진행 상태인지는 알 수 없었습니다.

적용.

  • 탭 버튼과 패널을 role·aria-selected·aria-controls·aria-labelledby로 상호 연결
  • progressbar에 접근 가능한 이름과 값 속성 부여
  • 장식용 아이콘은 aria-hidden·focusable="false"로 낭독 대상에서 제외
// 설명용 예시: 실제 컴포넌트 이름이 아님
<button
  id={`tab-${key}`}
  role="tab"
  aria-selected={selected}
  aria-controls={`panel-${key}`}
>
  {label}
</button>

<section
  id={`panel-${key}`}
  role="tabpanel"
  aria-labelledby={`tab-${key}`}
>
  {children}
</section>
// 장식용 아이콘은 낭독·포커스 대상에서 제외
<svg aria-hidden="true" focusable="false" viewBox="0 0 24 24">
  <path d="..." />
</svg>

결과. 탭을 전환할 때 현재 선택 상태와 대상 패널이 함께 전달되고, 장식 요소가 낭독 흐름을 끊지 않게 됐습니다.

3. 키보드 포커스 가시성 확보​

문제. 기본 아웃라인을 디자인상 제거한 컴포넌트가 있어, 키보드로 이동하면 지금 어디에 있는지 화면에서 확인할 수 없는 상태였습니다. 마우스를 쓰기 어려운 사용자에게는 조작 자체가 불가능해집니다.

적용.

  • 아웃라인을 없애는 대신 focus-visible 기반 포커스 링을 공통 스타일로 적용
  • 마우스 클릭 시에는 링이 뜨지 않고, 키보드 이동 시에만 노출되도록 분리
  • 링크·버튼 등 조작 가능한 요소 전반에 동일 기준 적용
// 설명용 예시: 실제 경로·토큰 이름이 아님
<Link
  href="/domains/sample"
  className="focus-visible:outline-none focus-visible:ring-2 focus-visible:ring-primary focus-visible:ring-offset-2"
>
  상세 정보 보기
</Link>

결과. 디자인을 해치지 않으면서 키보드 사용자의 현재 위치가 항상 보이게 됐고, 포커스 스타일 기준이 컴포넌트마다 달라지는 문제도 정리됐습니다.

4. 색 대비와 가독성 개선​

문제. 보조 정보를 옅은 회색으로 처리한 영역이 많아, 요약 카드·푸터·내비게이션의 텍스트 대비가 기준에 미치지 못했습니다. 저시력 사용자와 고령 사용자에게는 사실상 읽히지 않는 정보였습니다.

적용.

  • 저대비 텍스트 색상을 대비 기준에 맞춰 상향
  • 요약 카드, 푸터, 내비게이션 등 정보 밀도가 높은 영역 우선 정비
  • 색상만으로 상태를 구분하던 표현에 텍스트·아이콘 정보를 함께 제공

결과. "보조 정보라서 흐리게"라는 관성을 걷어내고, 화면에 표시된 정보는 실제로 읽을 수 있다는 기준을 맞췄습니다.

5. axe + Playwright 기반 접근성 회귀 테스트​

문제. 접근성은 한 번 고쳐도 다음 화면이 추가되면 다시 무너집니다. 수동 점검에 의존하면 정비한 기준이 유지되지 않습니다.

적용.

  • @axe-core/playwright를 E2E 테스트에 연결해 주요 경로를 자동 점검
  • 공개 페이지뿐 아니라 인증이 필요한 페이지까지 포함해 8개 경로 구성
  • 위반 항목이 발견되면 테스트가 실패하도록 해, 릴리즈 전에 걸리도록 구성
// 설명용 예시: 실제 테스트 경로가 아님
import AxeBuilder from "@axe-core/playwright";

const 점검경로 = ["/", "/domains", "/domains/sample", "/settings"];

for (const 경로 of 점검경로) {
  test(`접근성 점검: ${경로}`, async ({ page }) => {
    await page.goto(경로);

    const results = await new AxeBuilder({ page }).analyze();
    expect(results.violations).toEqual([]);
  });
}

결과. 접근성 점검이 감이나 개인 역량에 의존하지 않고 CI에서 반복 실행 가능한 검증이 됐습니다. 주요 공개·인증 경로 8개가 점검을 통과한 상태로 유지됩니다.

결과​

  • 문서 구조 정상화 — heading 계층 정리, 중복 main 제거, landmark 재정비
  • 상태 UI의 의미 전달 — 탭/패널 ARIA 관계 설계, progressbar 접근 가능한 이름 부여
  • 키보드 조작 보장 — focus-visible 기반 포커스 가시성 공통 적용
  • 판독성 개선 — 저대비 텍스트 정비로 저시력·고령 사용자 가독성 향상
  • 회귀 방지 체계 구축 — axe + Playwright 자동 점검으로 공개·인증 8개 경로 통과 유지
  • 기능 회귀 없이 정리 — UI 리팩토링과 접근성 개선을 병행하면서 동작 회귀 없이 마무리

이 작업의 핵심은 개별 화면을 고친 것이 아니라, 접근성을 사람이 기억해서 지키는 항목에서 자동으로 걸리는 기준으로 옮긴 것이었습니다. 같은 기준을 성능 최적화와 함께 운영하면서, 화면 책임과 UI 경계를 더 명확하게 정리할 수 있었습니다.
