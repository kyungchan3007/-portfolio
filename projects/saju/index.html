🔮 SAJU:ME

2026.03 – 2026.06 · FE 개발 · 배포

사용자의 출생 정보로 오늘의 운세, 정통사주, 궁합, 성향, 음식·위치 추천을 제공하는 개인화 사주 플랫폼입니다.
사용자 앱, 관리자 앱, 문서 사이트를 monorepo로 구성하고, BFF 보안 경계와 Cloudflare Edge 배포 환경을 구축했습니다.

문제 정의​

SAJU:ME는 사주 입력, 결과 조회, 광고/보호 게이트, 인증, 마이페이지, 관리자 기능이 함께 얽힌 서비스입니다. 클라이언트에서 백엔드 호출, 인증 쿠키 처리, 응답 파싱, UI 상태 처리가 섞이면 유지보수성과 보안성이 동시에 떨어질 수 있었습니다.

그래서 사용자 앱의 브라우저 요청은 BFF Route Handler를 통과하도록 정리하고, 인증·보안·응답 파싱은 서버 경계에서 처리했습니다. 운영 업무는 Admin Dashboard로 분리하고, 배포는 Cloudflare Workers 환경에 맞췄습니다.

  • 인증 토큰, CSRF, Turnstile 검증은 BFF API Guard에서 처리
  • 사용자별 결과 데이터는 authScope 기반 query key로 캐시 분리
  • 관리자 API 호출은 서버 fetch wrapper로 인증 헤더 주입 통일
  • SSR 배포는 Cloudflare Workers Edge Runtime에 맞게 구성
  • 변경 이력과 릴리즈 기준은 공통 버전 관리 전략으로 관리

주요 개선​

1. BFF 기반 인증/보안 경계 분리​

인증이 필요한 사주 결과 조회에서 토큰, CSRF, 봇 방어가 UI 코드와 섞이지 않도록 Next.js Route Handler를 BFF로 두었습니다. withApiGuards에서 CSRF와 Turnstile 검증을 강제하고, HttpOnly 쿠키의 access token은 서버에서만 읽도록 분리했습니다.

route.ts
export const POST = withApiGuards(
  { requireCsrf: true, requireTurnstile: true },
  async () => {
    const cookieStore = await cookies();
    const token = cookieStore.get(ACCESS_TOKEN_COOKIE_KEY)?.value;

    if (!token) {
      return NextResponse.json(createErrorResponse('LOGIN_REQUIRED'), {
        status: 401,
      });
    }

    const dailyResult = await onSajuDailyGetOnServer();
    return NextResponse.json({ success: true, data: dailyResult });
  },
);

결과: 클라이언트는 민감 정보를 직접 다루지 않고, 서버 경계에서 인증/보안 정책을 일관되게 적용했습니다.

2. 인증 범위 기반 React Query 캐시 분리​

사주 결과는 사용자별 민감 데이터입니다. 기능 단위 query key만 사용하면 로그인 사용자가 바뀌었을 때 이전 사용자의 결과 캐시가 재사용될 수 있어, authScope를 query key에 포함했습니다.

useSajuResultQuery.ts
export function useSajuResultQuery(returnTo: string) {
  const authScope = useAuthScope();

  return useQuery({
    queryKey: [...SAJU_RESULT_QUERY_KEY, authScope],
    queryFn: fetchSajuResultOnClient,
    staleTime: 60 * 60 * 1000,
    gcTime: 60 * 60 * 1000,
    retry: 1,
  });
}

결과: React Query 캐시를 유지하면서도 사용자별 결과 데이터가 섞일 가능성을 줄였습니다.

3. Admin Dashboard와 Edge 배포​

운영팀이 수작업으로 확인하던 사용자, 커뮤니티, 알림, 모니터링을 Admin Dashboard로 통합했습니다. 관리자 API는 서버 fetch wrapper에서 쿠키 기반 토큰 주입과 no-store 정책을 통일해 권한 경계를 관리했습니다.

authenticatedBackendFetch.ts
export async function authenticatedBackendFetch(path: string, init: RequestInit = {}) {
  const cookieStore = await cookies();
  const accessToken = cookieStore.get(ACCESS_TOKEN_COOKIE_KEY)?.value;

  const headers = new Headers(init.headers);
  headers.set('Authorization', `Bearer ${accessToken}`);

  const response = await fetch(`${BACKEND_API_BASE_URL}${path}`, {
    ...init,
    cache: 'no-store',
    headers,
  });

  return { success: response.ok, response };
}

Next.js SSR은 OpenNext.js와 nodejs_compat를 사용해 Cloudflare Workers 환경에 배포했습니다.

wrangler.toml — Before
# Workers 런타임에서 Node.js 모듈 미지원
[env.production]
routes = [
  { pattern = "example.com", zone_id = "..." }
]

# Next.js 의존 모듈 중 일부가 Node.js API 사용 불가
wrangler.toml — After
# nodejs_compat 플래그로 Node.js 환경 에뮬레이션
[env.production]
compatibility_flags = ["nodejs_compat"]
routes = [
  { pattern = "saju.me", zone_id = "..." }
  { pattern = "docs.saju.me", zone_id = "..." }
]

구현 방식:

  • OpenNext.js: Next.js 프로젝트를 Cloudflare Workers 호환 번들로 변환
  • nodejs_compat: Edge Runtime에서 crypto, stream 등 Node.js 표준 라이브러리 모듈 지원
  • 다중 도메인: 사용자 앱, 관리자 앱, 문서 사이트를 도메인별 분리 배포
  • CAPTCHA: Cloudflare Turnstile로 봇 공격 차단

결과: 관리자 API 인증 경계를 통일하고, 사용자 앱/admin/docs를 Cloudflare Workers 기반으로 분리 운영할 수 있게 했습니다.

결과​

  • BFF Route Handler에서 인증, CSRF, Turnstile 검증을 처리해 클라이언트 보안 책임 축소
  • authScope 기반 query key로 사용자별 사주 결과 캐시 분리
  • Admin Dashboard와 서버 fetch wrapper로 관리자 API 인증 경계 통일
  • Cloudflare Edge Runtime 기반으로 사용자 앱, 관리자 앱, 문서 사이트 분리 배포
  • 변경 이력과 릴리즈 판단 기준은 공통 버전 관리 전략으로 분리해 프로젝트별 중복 기준을 제거
