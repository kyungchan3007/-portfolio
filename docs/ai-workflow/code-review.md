#  병합 전 리뷰를 표준화한 이유

부족한 인력으로 빠르게 개발하다 보면, 리뷰가 사람의 컨디션과 시간에 따라 들쭉날쭉해지기 쉽습니다. 저는 이 변동성을 없애기 위해 **Claude 구현 → Codex 검증·수정** 리뷰 절차를 표준화했습니다. 누가 언제 리뷰하든 같은 기준으로, 같은 형식으로 판정되게 만드는 것이 목표였습니다.

## 기본 절차

```
① Claude가 구현 완료
      ↓
② Codex에 검증 요청
      ↓
③ 품질 게이트로 작업 유형 분류
      ↓
④ 검증 기준에 따라 리뷰
      ↓
   ┌──┴──┐
  PASS  HOLD
   ↓      ↓
  병합   Codex가 직접 수정 → 재검증
```

## 게이트 매트릭스

작업 성격에 따라 검토 범위를 먼저 나누고, 그다음 단계별 게이트를 적용하는 패턴을 사용했습니다. 세부 기준을 모두 공개하기보다, 작업 유형에 맞는 검토 축을 고정해 리뷰 편차를 줄이는 데 목적을 뒀습니다.

### 유형 분류 방식

| 분류 패턴 | 설명 |
|---|---|
| 기능 변경 | 비즈니스 로직과 상태 흐름 중심으로 검토합니다. |
| 인증·보안 변경 | 접근 제어와 민감정보 처리 중심으로 검토합니다. |
| UI·표현 변경 | 렌더링 영향 범위와 회귀 여부 중심으로 검토합니다. |
| 운영·배포 변경 | 실행 조건과 운영 안정성 중심으로 검토합니다. |

### 게이트 적용 방식

G1~G5처럼 단계를 나눈 게이트 패턴을 사용했습니다. 각 단계는 로직, 계약, 보안, 상태 일관성, 회귀 위험처럼 서로 다른 관점을 담당하도록 구성했고, 작업 유형에 따라 필요한 단계만 조합해 검토했습니다.

## 리뷰 출력 형식

리뷰 결과가 매번 다른 모양으로 나오지 않도록, 근거와 판단이 한눈에 보이는 고정된 출력 패턴을 사용했습니다. 세부 템플릿을 공개하기보다, 같은 형식으로 기록하고 같은 기준으로 판정하는 흐름을 유지하는 데 중점을 뒀습니다.



## 재검증 방식

HOLD가 나온 뒤 수정해 다시 요청하면, 전체를 처음부터 반복하기보다 **변경된 범위부터 우선 검증**하도록 기준을 잡았습니다. 재검증 시간을 줄이되, 이전 Critical/High 이슈가 남아 있으면 반드시 다시 확인하도록 했습니다.

## 성능 리뷰

기능 리뷰가 코드 diff를 보는 것과 달리, 성능은 실제로 배포본을 띄워 측정하고 원인을 찾아 고치는 반복 작업입니다. 이 판단도 사람이 리포트를 눈으로 훑으면 회차마다 달라지기 때문에, 측정 결과를 리뷰 입력으로 고정하고 판단 기준을 미리 정해 둔 상태에서 돌렸습니다.

`Lighthouse CLI로 리포트 생성 → 리뷰 입력 → 병목 분류 → 위험도 순 대응 → 적용 → 재측정 → 회귀 검증`

```bash
npx lighthouse http://localhost:3000 \
  --preset=desktop \
  --output=html \
  --output-path=./lighthouse-report.html \
  --no-enable-error-reporting
```

판단이 흔들리지 않도록 성능 지침서를 별도로 정리해 Skill로 연결했습니다. 다루는 범위는 아래와 같습니다.

| 단계 | 항목 |
|---|---|
| 기준 정의 | 핵심 원칙 · 지표 선정 · 기준값 · 페이지 유형별 우선순위 |
| 측정 | 확인 순서 · 측정 절차 · 계측 코드 원칙 |
| 판단 | 병목 분류 기준 · 작업 우선순위(위험도) · 금지 패턴 |
| 구현 | 공통 번들 · 지연 로드 · 이미지 · CSS · JS/React · 폰트 |
| 설계 | UX/UI 유의점 · 비즈니스 로직 유의점 · 전역 구조 변경 규칙 |
| 검증 | 최소 검증 · 구조 변경 시 필수 · 전역 변경 시 추가 · 개선 후 확인 |
| 기록 | 성능 개선 서술 원칙 · 최종 체크리스트 |

총 21개 항목이며, 세부 규칙은 사내 프로젝트에 적용 중이라 공개하지 않습니다.

핵심은 **작업을 위험도 낮은 순서로 강제한 것**이었습니다. 전역 Provider나 레이아웃 구조는 효과가 가장 크지만 인증·알림 같은 공통 기능 회귀 위험도 가장 커서, 앞 단계를 모두 거친 뒤에만 손대도록 순서를 고정했습니다.

실제 적용 결과는 [성능 최적화](../projects/performance.md)에 정리했습니다.
