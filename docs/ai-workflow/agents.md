# 에이전트 역할 분리

작업을 **구현 → 검증 → 기록** 3단계로 나누고, 각 단계를 다른 에이전트가 담당합니다.

## 🔨 Implementer (Claude)

**역할:** 기능 구현 전용

- 스킬 파일을 읽고 도메인 규칙을 파악한 뒤 코드를 작성합니다
- 구현 완료 후 diff를 Reviewer에게 넘깁니다
- 코드 수정은 이 에이전트만 수행합니다

```
담당: 기능 추가, 버그 수정, 리팩터링
금지: 자신이 만든 코드를 스스로 최종 검증
```

## 🔍 Reviewer (Codex)

**역할:** 검증 전용 — 코드 수정 없음

- Claude가 넘긴 diff를 받아 리뷰합니다
- 판정은 반드시 `MERGE: PASS` 또는 `MERGE: HOLD` 둘 중 하나
- 스타일 코멘트가 아닌 **실제 리스크**만 지적합니다

```
담당: 비즈니스 로직 무결성, 보안, 회귀 위험, API 계약
금지: 코드 직접 수정, 추측성 지적
```

### 리뷰 우선순위

```
1순위  비즈니스 로직 무결성
2순위  API 계약 / 타입 안전성
3순위  인증 / 보안
4순위  상태관리 / 캐시 일관성
5순위  회귀 위험 / 운영 안정성
```

### 병합 게이트

| 상황 | 판정 |
|------|------|
| Critical 1개 이상 | 🔴 `MERGE: HOLD` |
| High 미해결 1개 이상 | 🔴 `MERGE: HOLD` |
| Medium / Low만 남은 경우 | 🟡 리스크 명시 후 조건부 `MERGE: PASS` |
| 이슈 없음 | 🟢 `MERGE: PASS` |

## 📝 Worklog

**역할:** 변경 기록 전용

- PASS 판정 이후 변경 내용을 요약합니다
- 프로젝트 관리 시스템에 주간 작업 기록을 반영합니다
- 누락 없이 기록하는 것이 목표

```
담당: 변경 요약, 작업 기록, 주간 worklog
금지: 구현 판단, 코드 수정
```

## 💡 역할 분리의 효과

같은 AI가 구현과 검증을 동시에 하면 **자신이 만든 코드에서 맹점**이 생겨요. 실제로 AI가 잘못 구현한 코드를 스스로 리뷰하면서 "문제없다"고 판정하는 경우가 종종 있습니다.

구현체와 리뷰어를 다른 AI로 분리하면:
- 🎯 리뷰 독립성 확보
- 🛡️ 회귀/보안 리스크 감소
- 📋 책임 경계 명확화
