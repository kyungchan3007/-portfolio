# 왜 구현과 검증을 다른 AI에게 맡겼나

AI에게 일을 맡기며 가장 크게 데인 지점 중 하나는, **AI가 자기 코드를 스스로 리뷰하면 문제를 못 본다는 것**이었습니다. 잘못 구현한 코드를 같은 AI가 검토하면서 태연하게 "문제없다"고 판정하는 일이 반복됐습니다.

그래서 작업을 **구현 → 검증 → 기록** 3단계로 나누고, 각 단계를 다른 에이전트에게 맡겼습니다.

## 🔨 Implementer (Claude) — 구현만 한다

- 스킬 파일을 읽고 도메인 규칙을 파악한 뒤 **초안 코드를 작성**하게 했습니다
- 구현이 끝나면 diff를 Reviewer에게 넘기도록 했습니다
- 넘긴 뒤의 검증과 수정에는 관여하지 않도록 역할을 좁혔습니다

```
담당: 기능 구현 (초안 작성)
금지: 자신이 만든 코드를 스스로 검증·교정
```

## 🔍 Reviewer (Codex) — 검증하고, 직접 고친다

구현 AI와 **다른 AI**에게 검증과 교정을 맡겨, 자기 코드에 대한 맹점을 없앴습니다. Codex는 판정만 하는 게 아니라 **잘못된 코드와 구조를 직접 수정**했습니다.

- Claude가 넘긴 diff를 받아 리뷰하게 했습니다
- 판정은 반드시 `MERGE: PASS` 또는 `MERGE: HOLD` 둘 중 하나로 강제했습니다
- 잘못된 코드·구조·로직은 **Codex가 직접 고쳐** 기준을 통과시키도록 했습니다
- 스타일 취향이 아니라 **실제 리스크**를 기준으로 판단하도록 규칙을 박아 넣었습니다

```
담당: 검증(비즈니스 로직·보안·회귀·API 계약) + 잘못된 코드·구조 직접 수정
금지: 추측성 지적
```

리뷰 우선순위도 고정해, 매번 같은 기준으로 보게 만들었습니다.

```
1순위  비즈니스 로직 무결성
2순위  API 계약 / 타입 안전성
3순위  인증 / 보안
4순위  상태관리 / 캐시 일관성
5순위  회귀 위험 / 운영 안정성
```

병합 여부도 판정 규칙으로 못 박았습니다.

| 상황 | 판정 |
|------|------|
| Critical 1개 이상 | 🔴 `MERGE: HOLD` |
| High 미해결 1개 이상 | 🔴 `MERGE: HOLD` |
| Medium / Low만 남은 경우 | 🟡 리스크 명시 후 조건부 `MERGE: PASS` |
| 이슈 없음 | 🟢 `MERGE: PASS` |

## 📝 Worklog — 기록만 한다

- PASS 판정 이후 변경 내용을 요약하게 했습니다
- 프로젝트 관리 시스템에 주간 작업 기록을 반영하도록 했습니다
- 누락 없이 남기는 것을 목표로 삼았습니다

```
담당: 변경 요약, 작업 기록, 주간 worklog
금지: 구현 판단, 코드 수정
```

## 이렇게 나누고 나서

구현은 Claude, 검증과 교정은 Codex로 분리하자, 부족한 인력을 메우면서도 리뷰의 독립성을 확보할 수 있었습니다.

- 🎯 리뷰 독립성 확보 — 자기 코드를 봐주는 게 아니라 남의 코드를 보고 고친다
- 🛡️ 회귀·보안 리스크 감소 — 판정 기준이 흔들리지 않는다
- 📋 책임 경계 명확화 — Claude는 구현, Codex는 검증·수정으로 나뉜다
