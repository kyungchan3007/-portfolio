# 스킬 시스템 — 지침을 온톨로지로 구조화하다

토큰 낭비 문제를 정면으로 풀기 위해 만든 것이 스킬 시스템입니다. 핵심 원칙은 하나였습니다 — **지금 이 작업에 필요한 문서만 골라 읽는다.**

## 왜 스킬로 나눴나

AI에게 모든 프로젝트 규칙을 매번 대화에서 설명하면 토큰이 순식간에 새어 나갔습니다. 그래서 규칙을 도메인별 지침 파일로 미리 정리해 두고, 작업에 맞는 파일만 읽게 만들었습니다.

## 파일 구조

```
.agents/skills/{skill-name}/
├── SKILL.md          ← 진입점. 읽기 규칙 + 공통 원칙
└── references/       ← 도메인별 세부 지침
    ├── architecture.md
    ├── api.md
    ├── auth.md
    └── ...
```

## "필요한 것만" 읽는 규칙

`SKILL.md`에 도메인별 분기 규칙을 넣어, 작업 종류에 따라 읽을 파일을 좁혔습니다.

```
- 입력 폼 흐름 작업이면 → references/form-flow.md 읽기
- API 연동 작업이면       → references/api.md 읽기
- UI 스타일 작업이면      → references/ui-style.md 읽기
```

한 도메인 작업에서 읽는 파일은 `SKILL.md + reference 1개`를 원칙으로 잡았습니다. 여러 도메인에 걸칠 때만 추가로 읽게 했습니다. 이 원칙 하나로 컨텍스트가 부풀던 문제를 눌렀습니다.

## 구현 전 필수 출력을 강제했다

스킬을 읽은 뒤 AI가 코드 작성 전에 반드시 아래 형식을 출력하도록 만들었습니다.

```
읽은 문서: SKILL.md, references/XXX.md
요약 (5줄 이내): ...
파일 레이어 계획:
  - model/: 상수, 순수 유틸, 데이터 모델 변환
  - hooks/: useQuery, useMutation, 상태·플로우 로직
  - ui/: props 받아 렌더링만
  - entities/: API 타입, 서버/클라이언트 통신 경계
  - app/api/: BFF route
```

이 출력을 강제하니, AI가 코드를 쓰기 전에 레이어 배치를 한 번 더 검토하게 되어 엉뚱한 방향으로 새는 일이 크게 줄었습니다.

## 운영 중인 스킬 목록

| 스킬 | 용도 |
|------|------|
| `frontend` | 웹 애플리케이션 전체 도메인 구현 |
| `code-review-guard` | 병합 전 리뷰 |
| `code-review-workflow` | Claude → Codex 검증 절차 |
| `design-system-hybrid` | 디자인 시스템 운영 |
| `weekly-worklog` | 주간 작업 기록 |
| `prompt-trigger-eval` | 스킬 트리거 평가 |
| `change-summary-report` | 변경 요약 리포트 |
| `best-practices` | 웹 성능/렌더링 점검 |
| `web-design-guidelines` | 웹 UI 가이드 점검 |

## 스킬은 결국 온톨로지 노드였다

만들다 보니, 스킬은 단순한 지침 파일이 아니라 작업 도메인과 지식을 구조화하는 **온톨로지 노드**라는 걸 알게 됐습니다.

- **`SKILL.md`** — 도메인 진입점. "어떤 작업일 때 어떤 reference를 읽어라"는 분기 규칙.
- **`references/`** — 세부 지식 그래프. 도메인별 규칙·경계·주의사항을 분리 보관.
- **`description`** — 트리거 분류 기준. 요청을 이 스킬로 라우팅할지 판단하는 문장.

좋은 `description`은 **"언제 써야 하는지"** 와 **"언제 쓰면 안 되는지"** 를 함께 명시해야 한다는 걸 경험으로 배웠습니다.

```yaml
description: >-
  웹 애플리케이션 구현 또는 변경 요청에 사용한다.
  라우트, API 연동, 상태관리 요청이 해당된다.
  백엔드 전용 변경, 인프라 전용 작업에는 사용하지 않는다.
```

"쓰면 안 되는 경우"를 빼먹으면 범위가 너무 넓어져 관련 없는 요청에서도 발동(오탐)했습니다.

## 경계는 감이 아니라 테스트로 다듬었다

`description`이 부정확하면 엉뚱한 스킬이 발동하거나, 발동해야 할 스킬이 침묵했습니다. 그래서 `prompt-trigger-eval`로 `should_trigger` / `should_not_trigger` 샘플을 돌려 발동 정확도를 검증했습니다.

```
should_trigger 예시:
- "결과 페이지에서 캐시가 꼬여서 이전 데이터가 보여요. 상태관리 점검해줘."

should_not_trigger 예시:
- "백엔드 사용자 인증 엔진의 토큰 검증 로직만 수정해줘."
```

검증 결과를 `references/test-results.md`에 누적하면서 어떤 스킬이 어떤 경계에서 오탐/미탐을 냈는지 관리했고, 문제가 보이면 `description`을 고친 뒤 같은 세트로 다시 돌렸습니다. 이 반복이 완성율을 95%까지 끌어올린 실질적인 작업이었습니다.

:::tip
`prompt-trigger-eval`은 상시 실행하는 스킬이 아닙니다. 스킬 경계를 점검하거나 `description`을 조정할 때만 쓰는 **평가 전용 스킬**입니다.
:::

## 스킬은 살아있는 문서다

> 새 라우트, 새 도메인, 반복되는 실수, 중요한 규칙이 생기면 스킬 파일을 함께 갱신했습니다.

스킬 파일을 코드와 동기화된 살아있는 문서로 다뤘습니다. 프로젝트가 바뀌면 스킬도 함께 바꿨습니다.
