# 스킬 시스템 — 지침을 온톨로지로 구조화하다

토큰 낭비 문제를 정면으로 풀기 위해 만든 것이 스킬 시스템입니다. 핵심 원칙은 하나였습니다 — **지금 이 작업에 필요한 문서만 골라 읽는다.**

## 왜 스킬로 나눴나

AI에게 모든 프로젝트 규칙을 매번 대화에서 설명하면 토큰이 순식간에 새어 나갔습니다. 그래서 규칙을 도메인별 지침 파일로 미리 정리해 두고, 작업에 맞는 파일만 읽게 만들었습니다.

일반적인 패턴을 사용했고, 프로젝트 특성에 맞게 재구성하여 적용했습니다.

## "필요한 것만" 읽는 규칙

도메인별 분기 규칙을 두어, 작업 종류에 따라 필요한 문서만 읽도록 범위를 좁혔습니다.

```
- 입력 흐름 작업이면 → 관련 흐름 문서 확인
- API 연동 작업이면  → 계약/통신 규칙 문서 확인
- UI 작업이면        → 표현 규칙 문서 확인
```

한 작업에서 읽는 문서는 최소한으로 제한하고, 여러 도메인에 걸칠 때만 추가 문서를 읽도록 했습니다. 이 원칙 하나로 컨텍스트가 부풀던 문제를 줄일 수 있었습니다.

## 구현 전 필수 출력을 강제했다

스킬을 읽은 뒤 AI가 코드 작성 전에 반드시 아래 형식을 출력하도록 만들었습니다.

```md
읽은 문서: 관련 지침 문서
요약 (5줄 이내): ...
구현 계획:
- 어떤 레이어에 둘지
- 어떤 검증이 필요한지
- 무엇을 먼저 확인할지
```

이 출력을 강제하니, AI가 코드를 쓰기 전에 레이어 배치를 한 번 더 검토하게 되어 엉뚱한 방향으로 새는 일이 크게 줄었습니다.

## 역할별로 나눠 운영했다

| 역할 | 설명 |
|------|------|
| 구현 | 기능 구현에 필요한 문서와 규칙을 먼저 읽고 초안을 작성 |
| 검증 | 병합 전 코드와 구조를 점검하고 수정 방향을 정리 |
| 디자인/표현 기준 | UI와 표현 계층 규칙을 확인 |
| 기록 | 작업 결과와 변경 사항을 정리 |
| 평가 | 트리거와 규칙이 제대로 동작하는지 점검 |

## 스킬은 결국 온톨로지 노드였다

만들다 보니, 스킬은 단순한 지침 파일이 아니라 작업 도메인과 지식을 구조화하는 **온톨로지 노드**라는 걸 알게 됐습니다.

- **진입 문서** — 어떤 작업일 때 어떤 지침을 먼저 읽어야 하는지 정하는 규칙
- **세부 지침 문서** — 도메인별 규칙·경계·주의사항을 정리한 문서
- **트리거 문장** — 요청을 어떤 작업 유형으로 분류할지 판단하는 기준

좋은 `description`은 **"언제 써야 하는지"** 와 **"언제 쓰면 안 되는지"** 를 함께 명시해야 한다는 걸 경험으로 배웠습니다.

```yaml
description: >-
  웹 애플리케이션 구현 또는 변경 요청에 사용한다.
  라우트, API 연동, 상태관리 요청이 해당된다.
  백엔드 전용 변경, 인프라 전용 작업에는 사용하지 않는다.
```

"쓰면 안 되는 경우"를 빼먹으면 범위가 너무 넓어져 관련 없는 요청에서도 발동(오탐)했습니다.

## 경계는 감이 아니라 테스트로 다듬었다

트리거 문장이 부정확하면 엉뚱한 작업 분류가 일어나거나, 필요한 규칙이 누락됐습니다. 그래서 발동해야 하는 경우와 발동하지 말아야 하는 경우를 나눠 샘플 기반으로 검증했습니다.

```
should_trigger 예시:
- "결과 페이지에서 캐시가 꼬여서 이전 데이터가 보여요. 상태관리 점검해줘."

should_not_trigger 예시:
- "백엔드 사용자 인증 엔진의 토큰 검증 로직만 수정해줘."
```

검증 결과를 누적하면서 어떤 경계에서 오탐/미탐이 났는지 관리했고, 문제가 보이면 트리거 문장을 고친 뒤 같은 세트로 다시 돌렸습니다. 이 반복이 완성율을 95%까지 끌어올린 실질적인 작업이었습니다.

:::tip
이 평가는 상시 실행하는 작업이 아니라, 경계를 점검하거나 트리거 문장을 조정할 때만 수행하는 평가 절차였습니다.
:::

## 스킬은 살아있는 문서다

> 새 라우트, 새 도메인, 반복되는 실수, 중요한 규칙이 생기면 관련 지침 문서를 함께 갱신했습니다.

지침 문서는 코드와 동기화된 살아있는 문서로 다뤘습니다. 프로젝트가 바뀌면 기준 문서도 함께 바꿨습니다.
