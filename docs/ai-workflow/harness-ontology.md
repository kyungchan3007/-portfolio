# 하네스 + 온톨로지 엔지니어링

AI를 개발에 도입했지만 오히려 느려졌던 문제를, 저는 **하네스**와 **온톨로지**를 직접 설계하는 것으로 풀었습니다. 이 문서는 그 과정을 정리한 기록입니다.

---

## 배경 — 왜 직접 엔지니어링까지 했나

처음엔 AI에게 그냥 일을 맡기면 될 줄 알았습니다. 하지만 지침 없는 AI는 관련 없는 파일을 읽어 토큰을 태우고, 프로젝트 규칙을 벗어난 코드를 내놓고, 그걸 되돌리는 데 시간을 더 썼습니다. 마감을 당기려고 도입한 AI가 오히려 발목을 잡는 상황이었습니다.

그래서 "AI를 어떻게 쓰느냐"를 하나의 엔지니어링 문제로 놓고 공부하기 시작했습니다. 핵심 질문은 두 가지였습니다.

- **어떻게 하면 매번 같은 기준으로 동작하게 만들까** → 하네스
- **어떻게 하면 필요한 지식만 골라 읽게 만들까** → 온톨로지

### 95% 완성율까지 — 실험과 튜닝의 반복

이 구조를 프로덕션에 바로 적용하지 않았습니다. **사이드 프로젝트에서 먼저 실험**하며, 작업 완성율이 95% 이상으로 올라갈 때까지 다음 루프를 반복했습니다.

```
① prompt-trigger-eval 로 스킬 트리거 정확도 점검 (오탐/미탐 찾기)
        ↓
② 스모크 테스트로 실제 동작 확인
        ↓
③ 결과를 보고 지침서(SKILL.md / references) 수정
        ↓
④ 스킬·도메인 경계선을 어떻게 명확히 나눌지 다시 설계
        ↓
   (완성율 95% 이상이 될 때까지 반복)
```

가장 오래 붙잡았던 건 **경계선**이었습니다. 스킬의 담당 범위가 조금만 넓어도 엉뚱한 요청에서 발동했고, 조금만 좁아도 발동해야 할 때 발동하지 않았습니다. `prompt-trigger-eval`로 오탐/미탐을 수치로 확인하고, 지침서를 고치고, 다시 검증하는 과정을 거치며 경계를 날카롭게 다듬어 갔습니다. 그 결과 "어떤 상황엔 어떤 스킬, 어떤 수정엔 어떤 검증, 어떤 작업엔 어떤 구현 스킬, 수정 범위에 따라 검증 단계를 어떻게 나눌지"를 규칙으로 굳힐 수 있었습니다.

## 하네스 — 실행을 고정하다

하네스는 AI가 매번 자유롭게 판단하지 않도록 **실행 순서, 역할, 출력 형식을 사전에 고정**한 운영 장치입니다.

| 고정 항목 | 내용 |
|---|---|
| **역할** | `implementer`, `reviewer`, `worklog` |
| **절차** | 구현 → 검증 → 기록 순서 준수 |
| **판정** | `MERGE: PASS` 또는 `MERGE: HOLD` 둘 중 하나만 허용 |
| **출력** | 리뷰 결과 형식, 변경 요약 형식, 기록 형식 고정 |

하네스가 없을 때 AI는 요청마다 다르게 동작했습니다. 같은 작업인데 검증 기준이 달라지고 출력 형식이 제각각이었습니다. 이 변동성을 없애기 위해 절차와 형식을 명시적 규칙으로 박아 넣었습니다.

:::note
하네스는 AI가 완전 자동으로 실행하는 파이프라인이 아닙니다. 명시적 운영 규칙과 에이전트 지침에 따라 작동하는 구조로 설계했습니다.
:::

## 온톨로지 — 지식을 분류하다

온톨로지는 **작업 도메인과 필요한 지식을 분류하는 체계**입니다. 어떤 요청이 들어왔을 때 어떤 스킬을 쓰고 어떤 참조 문서를 읽어야 하는지를 미리 정의해, AI가 매번 처음부터 헤매지 않게 만들었습니다.

### 도메인을 스킬로 나눴다

| 스킬 | 담당 도메인 |
|---|---|
| `frontend` | 웹 애플리케이션 전체 (라우트, API 연동, 상태관리, BFF) |
| `documentation` | 문서 사이트 운영 |
| `design-system-hybrid` | 디자인 시스템 / 토큰 / 컴포넌트 |
| `code-review-workflow` | 검증 절차 및 게이트 매트릭스 |
| `code-review-guard` | 병합 전 코드 리뷰 |
| `weekly-worklog` | 작업 기록 |

### 필요한 지식만 읽게 만들었다

각 스킬을 **도메인 진입점(`SKILL.md`)** 과 **세부 지식 그래프(`references/`)** 로 구성했습니다.

```
.agents/skills/frontend/
├── SKILL.md                  ← 도메인 진입점 + 읽기 규칙
└── references/
    ├── architecture.md       ← 전체 구조/레이어 경계
    ├── api.md                ← API 연동/BFF 경계
    ├── auth.md               ← 인증/로그인 흐름
    ├── feature-flows.md      ← 주요 기능 흐름
    ├── infrastructure.md     ← 배포/인프라
    └── ...
```

`SKILL.md` 안에 "이 작업이면 이 reference를 읽어라"는 분기 규칙을 넣어, 작업에 필요한 파일만 선택적으로 읽도록 했습니다. 토큰 낭비 문제를 여기서 정면으로 해결했습니다.

### 트리거 문장 — 라우팅의 기준을 잡았다

각 스킬 `SKILL.md`의 frontmatter `description`은 단순 설명이 아니라, **사용자 요청을 어떤 도메인으로 분류할지 결정하는 트리거 문장**입니다.

```yaml
---
name: frontend
description: >-
  웹 애플리케이션 구현 또는 변경 요청에 사용한다.
  라우트, API 연동, 상태관리, UI 구성 요청이 해당된다.
  백엔드 전용 변경, 인프라 전용 작업에는 사용하지 않는다.
---
```

이 문장의 품질이 스킬 라우팅 정확도를 좌우했고, 95% 완성율을 만드는 핵심 지렛대였습니다.

## 하네스와 온톨로지가 함께 도는 흐름

```
사용자 요청
    │
    ▼
[온톨로지] skill description 매칭
    │   → 어떤 스킬인가? (frontend / design-system-hybrid / ...)
    │
    ▼
[온톨로지] SKILL.md 읽기 규칙으로 필요한 reference 선택
    │   → 어떤 지식이 필요한가? (api.md / auth.md / feature-flows.md / ...)
    │
    ▼
[하네스] implementer 역할 수행
    │   → 구현 전 필수 출력 (읽은 문서, 레이어 계획)
    │   → 코드 작성
    │
    ▼
[온톨로지] gate-matrix로 작업 유형 분류
    │   → 어떤 검증 조합인가?
    │
    ▼
[하네스] reviewer 역할 수행
    │   → code-review-guard + 선택 스킬 적용
    │   → MERGE: PASS / MERGE: HOLD 판정
    │
    ▼
[하네스] worklog 역할 수행
        → 변경 요약 + 작업 기록
```

## 실제로 이렇게 갈렸다

경계와 규칙을 다듬은 결과, 작업 유형별로 스킬·검증이 이렇게 자동으로 결정됩니다.

### 사용자 데이터 조회 오류 대응

- **선택되는 스킬:** `frontend`
- **선택되는 reference:** `references/feature-flows.md`
- **게이트:** 데이터 조회 유형 → `frontend`, `code-review-guard` 필수 / `best-practices` 선택
- **검증 포커스:** G1(비즈니스 로직), G2(API 계약), G4(상태관리/캐시), G5(회귀)
- **검증 명령:** 데이터 route/client query unit + `typecheck`

### React Query 캐시/무효화 변경

- **선택되는 스킬:** `frontend`
- **선택되는 reference:** `references/feature-flows.md` 또는 관련 도메인 reference
- **게이트:** React Query 캐시/무효화 유형 → `frontend`, `code-review-guard` 필수 / `best-practices` 선택
- **검증 포커스:** G1, G2, G4, G5
- **검증 명령:** query key/invalidation unit + `typecheck`

### CI/CD 워크플로우 변경

- **선택되는 스킬:** `frontend`
- **선택되는 reference:** `references/infrastructure.md`
- **게이트:** CI/CD 워크플로우 유형 → `frontend`, `code-review-guard` 필수
- **검증 포커스:** G1(로직), G3(보안), G4(상태), G5(회귀)
- **검증 명령:** YAML diff, 트리거/secret/branch 조건 확인

### 디자인 시스템/토큰 변경

- **선택되는 스킬:** `design-system-hybrid`, `code-review-guard`
- **선택되는 reference:** 디자인 시스템 관련 문서
- **게이트:** 디자인 시스템/토큰 유형 → `design-system-hybrid`, `code-review-guard` 필수 / `web-design-guidelines` 선택
- **검증 포커스:** G1, G2, G4, G5 + 접근성
- **검증 명령:** Storybook build + `typecheck`

## 트리거 품질 검증 — `prompt-trigger-eval`

95% 완성율을 만든 도구가 바로 이것입니다. 스킬 라우팅은 `description` 문장의 품질에 달려 있는데, 그 품질을 감이 아니라 **수치로** 검증했습니다.

- **트리거 문장이 너무 넓으면** → 관련 없는 요청에서 잘못 발동(오탐, FP)
- **트리거 문장이 너무 좁으면** → 발동해야 할 요청에서 발동 실패(미탐, FN)

`prompt-trigger-eval`은 `should_trigger` / `should_not_trigger` 샘플 세트로 스킬 발동 정확도를 검증하는 **평가용 스킬**입니다.

| 평가 항목 | 의미 |
|---|---|
| **TP** (정탐) | 발동해야 하는 요청에서 정상 발동 |
| **TN** (정거부) | 발동하지 말아야 하는 요청에서 정상 미발동 |
| **FP** (오탐) | 발동하지 말아야 하는 요청에서 잘못 발동 |
| **FN** (미탐) | 발동해야 하는 요청에서 발동 실패 |

```
① 평가 대상 스킬 지정
② should_trigger / should_not_trigger 샘플 세트 준비
③ 분류 정확도 점검 (TP / TN / FP / FN 집계)
④ 오탐/미탐이 있으면 description 문장 조정
⑤ 같은 세트로 재검증 → 개선 여부 확인
⑥ 결과를 references/test-results.md에 누적
```

결과를 `test-results.md`에 쌓으면서 어떤 스킬이 어떤 경계에서 오탐/미탐을 냈는지 추적했습니다. 이 과정을 반복해 온톨로지의 도메인 경계를 계속 조정했고, 그렇게 완성율을 끌어올렸습니다.

:::tip 사용 시점
`prompt-trigger-eval`은 일반 구현 작업 중 상시 실행하는 스킬이 아닙니다. "프롬프트 테스트 하자", "트리거 검증해줘", "오탐/미탐 점검"처럼 스킬 경계를 점검할 때만 사용합니다.
:::

## 남은 한계

솔직하게, 이 구조도 완벽하지는 않습니다. 운영하며 마주한 한계는 다음과 같습니다.

- **description 품질 의존** — 트리거 문장이 부정확하면 오탐/미탐이 생깁니다. 주기적으로 `prompt-trigger-eval`로 점검해야 합니다.
- **문서 최신화 필요** — 코드가 바뀌면 `references/` 문서도 함께 갱신해야 합니다. 문서가 낡으면 잘못된 판단이 나옵니다.
- **완전 자동이 아님** — 명시적 운영 규칙과 에이전트 지침이 필요합니다. 절차는 사람이 의식적으로 따라야 작동합니다.
- **스킬 경계 관리 부담** — 프로젝트가 커질수록 경계를 잘 나눠야 오탐/미탐이 줄어듭니다.

그럼에도, 이 구조 덕분에 부족한 인력과 촉박한 마감 속에서도 품질 기준을 낮추지 않고 릴리즈할 수 있었습니다.
