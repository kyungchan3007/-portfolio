# AI Agent 운영 개요

프로덕션 개발에서 **Claude**와 **Codex** 두 AI를 역할 분리해서 개발 워크플로우에 통합하고 있습니다.

## 왜 도입했나

AI에게 작업을 맡길 때 생기는 가장 큰 문제는 **컨텍스트 낭비**예요.

아무 지침 없이 "이 기능 만들어줘" 하면 AI는 관련 없는 파일을 잔뜩 읽고, 불필요한 추상화를 만들고, 프로젝트 규칙을 무시한 코드를 짭니다. 토큰도 낭비되고 결과물 품질도 들쭉날쭉해요.

이 문제를 해결하기 위해 **스킬(Skill) 시스템**과 **에이전트 역할 분리**를 설계했습니다.

## 핵심 구조

```
.agents/
├── skills/     ← AI가 작업 전 읽는 도메인별 지침
└── agents/     ← 역할별 에이전트 프롬프트 (구현 / 검증 / 기록)
```

## 두 AI의 역할

| AI | 역할 | 특징 |
|----|------|------|
| **Claude** | 구현 | 코드 작성, 리팩터링, 기능 추가 |
| **Codex** | 검증 | diff 리뷰, PASS/HOLD 판정 |

구현과 검증을 같은 AI가 하면 자신이 만든 코드의 문제를 못 보는 경향이 있어요. 역할을 분리하면 독립적인 시각으로 리뷰가 가능합니다.

## 작업 흐름

```
1️⃣  Claude (구현)
    └─ 스킬 파일 읽기 → 도메인 규칙 파악 → 코드 작성

2️⃣  Codex (검증)
    └─ diff 수신 → 게이트 매트릭스 분류 → PASS / HOLD 판정

3️⃣  Worklog (기록)
    └─ 변경 요약 → 작업 기록 반영
```

HOLD가 나오면 Claude가 수정 → Codex 재검증 → PASS → 병합.

## 운영 모델 — 하네스 + 온톨로지

이 워크플로우는 스킬 시스템과 역할 분리를 넘어 **하네스 + 온톨로지** 구조로 작동합니다.

- **하네스**는 실행 절차(구현→검증→기록)와 판정 형식(`MERGE: PASS/HOLD`)을 고정해서 매번 같은 기준으로 동작하게 만듭니다.
- **온톨로지**는 작업 도메인을 분류하고 필요한 지식(`references/`)의 위치를 정의합니다. `SKILL.md`가 진입점이고 `references/`가 세부 지식 그래프입니다.
- 각 스킬의 `description`(트리거 문장)은 사용자 요청을 어떤 도메인으로 라우팅할지 결정합니다. 이 문장의 품질은 [`prompt-trigger-eval`](./harness-ontology#트리거-품질-검증--prompt-trigger-eval)로 오탐/미탐을 점검하며 지속 관리합니다.

자세한 내용은 [하네스 + 온톨로지 기반 AI Agent 운영](./harness-ontology) 문서를 참고하세요.
