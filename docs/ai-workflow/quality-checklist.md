# ✅ 서비스 품질 체크리스트

**FMS · BEMS · 원격 제어 · 프론트엔드 품질 검증**

개발이 "완료"된 시점에도 두 종류의 문제가 남아 있었습니다.

- **기능은 동작하지만 사용성이 나쁜 문제** — 행동 유도 버튼 위치, 카드·정보 배치,
  화면 전환 비용, 로딩 체감, 텍스트 가독성
- **기능 자체의 생존·회귀 문제** — 흰 화면, 새로고침 에러, 인증 흐름 누락

두 문제는 성격이 달라 검증 방식도 분리했습니다.
**UX/UI 품질**은 AI 에이전트 기반 진단으로 놓치기 쉬운 항목까지 끌어내 판단·수정했고,
**기능·비즈니스 로직**은 Smoke → Regression 순으로 확인했습니다.
개발 완료 시점을 **기준선**으로 두 트랙의 항목을 체크리스트로 고정하고
미충족 항목을 반복 리뷰로 해소하면서, 서비스 품질 체크리스트 충족률을 **50% → 80%** 로 끌어올렸습니다.

---

## 충족률 개선

![서비스 품질 체크리스트 충족률 50% → 80%](/img/quality-checklist-coverage.svg)

개발 완료만으로는 체크리스트의 절반 정도만 충족되는 경우가 많았습니다.
미충족 항목을 남겨두지 않고 반복 리뷰로 하나씩 해소하면서 충족률을 끌어올렸습니다.

---

## 검증 트랙 구분

| 트랙 | 무엇을 보는가 | 방법 |
| --- | --- | --- |
| **UX/UI 품질** | 기능은 되지만 쓰기 불편한 지점 | AI 에이전트 진단 → 개발자 판단·수정 |
| **기능·비즈니스 로직** | 앱 생존·인증·데이터 정합성 | Smoke → Regression 수동 검증 |

- **기준선 고정** — 개발 완료 시점에 확인해야 할 품질 항목을 체크리스트로 명문화
- **반복 리뷰** — 미충족 항목을 남기지 않고 해소될 때까지 재확인
- **동일 기준 적용** — 같은 체크리스트를 FMS · BEMS · 원격 제어 전반에 공통으로 사용

## 1. UX/UI 품질 체크리스트

기능이 동작해도 사용자가 불편을 느끼는 지점을 확인합니다.
AI 에이전트(UX/UI 품질 검사)로 화면별 문제 후보를 리스트업한 뒤,
**실제 사용자 영향과 수정 비용을 기준으로 개발자가 선별·수정**했습니다.

- **행동 흐름** — 주요 CTA 위치, 다음 행동의 명확성, 화면 전환 비용
- **정보 구조** — 카드·목록 배치와 정보 위계, 반응형에서의 위계 유지
- **체감 성능** — 로딩 대기 시간, 로딩 중 대기 경험(스켈레톤·점진 렌더)
- **가독성** — 텍스트 대비·크기, 여백·정렬 일관성, 접근성

---

## 2. 기능·비즈니스 로직 체크리스트

앱이 살아 있는지(Smoke)부터 확인한 뒤,
변경이 기존 기능을 망가뜨리지 않았는지(Regression) 확인합니다.

- **Smoke** — 첫 화면 렌더링, 내비게이션 이동, 새로고침 후 상태 유지, 로그인·로그아웃 흐름
- **Regression** — 변경 영향 범위, 반응형 레이아웃 유지, 데이터 정합성, 예외·빈 데이터 처리

---

## 결과

- 서비스 품질 체크리스트 충족률 **50% → 80%**
- **UX/UI 품질** — 행동 유도·정보 배치·로딩 체감·가독성 문제를 배포 전 단계에서 해소
- **기능·비즈니스 로직** — 흰 화면·새로고침 에러·인증 흐름 누락 등 반복 이슈 제거
- 같은 기준을 FMS · BEMS · 원격 제어에 공통 적용해 **프로젝트 간 품질 편차 축소**
