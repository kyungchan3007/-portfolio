🏷️ 버전 관리 — 릴리즈 표준화

FMS, BEMS, SAJU:ME에서 공통으로 적용한 Changesets 기반 릴리즈 관리 전략입니다.

목표는 단순히 버전을 올리는 것이 아니라, 소스코드 변경과 기본 검증이 끝난 뒤 PR을 올리기 전에 변경 의도와 영향 범위를 기록하고, 릴리즈 실행은 CI/CD가 일관되게 처리하도록 분리하는 것입니다.

왜 필요했나​

여러 프로젝트에서 기능 추가, 버그 수정, 리팩터링, 패키지 분리 작업이 동시에 진행되면 다음 문제가 반복됩니다.

문제영향
변경 이력이 PR 설명에만 남음릴리즈 후 어떤 변경이 포함됐는지 추적하기 어려움
패키지별 영향 범위가 불명확함monorepo 구조에서 불필요한 배포 또는 누락 발생
major/minor/patch 판단 기준이 사람마다 다름리뷰 기준이 흔들리고 breaking change가 숨겨질 수 있음
배포 실행 절차가 프로젝트마다 다름릴리즈 실수와 수동 작업 비용 증가

그래서 버전 관리를 개발자의 의도 기록과 자동화된 릴리즈 실행으로 나눴습니다.

적용 프로젝트​

프로젝트적용 목적
FMS도메인 기능 단위 변경 이력 추적, API 계약 변경 여부 명시
BEMS운영 화면과 데이터 수집/시각화 기능 변경 이력 관리
SAJU:MEweb, admin, ui, design-tokens 등 패키지별 릴리즈 범위 분리

프로젝트별 기술 스택과 배포 환경은 달라도, 릴리즈 판단 기준은 동일하게 유지했습니다.

기본 원칙​

핵심 원칙은 3가지입니다.

  1. PR 생성 전에 개발자가 semver 의도를 판단한다

    • patch: 버그 수정, 내부 안정화
    • minor: 하위 호환을 유지하는 기능 추가
    • major: 기존 사용 코드를 깨뜨리는 변경

  2. Changeset은 판단 결과를 기록한다

    • Changeset이 major/minor/patch를 자동 판단하지 않음
    • 개발자가 변경 의도와 영향 범위를 선택해 기록
    • PR에는 코드 변경과 Changeset이 함께 포함됨

  3. PR 리뷰에서 판단 근거를 검증한다

    • API 응답 구조가 바뀌었는가?
    • 기존 호출 코드 수정이 필요한가?
    • 외부 클라이언트 또는 다른 패키지에 영향이 있는가?

  4. 릴리즈 실행은 CI/CD가 담당한다

    • Changeset 파일 수집
    • package version 계산
    • changelog 생성
    • tag 생성 및 배포 워크플로우 실행

판단 기준​

Patch​

기존 사용 방식과 동작 계약을 유지하면서 오류를 수정하는 변경입니다.

// Patch: null 입력 처리 누락 수정
export function formatDate(value: string | null) {
  if (!value) return '-';
  return new Date(value).toLocaleDateString();
}

적용 기준:

  • 기존 API 시그니처 유지
  • 반환 타입 유지
  • 사용자가 기대하는 동작 유지
  • 버그 수정 또는 내부 안정화

Minor​

기존 사용 코드를 깨뜨리지 않고 기능을 추가하는 변경입니다.

// Minor: 선택 옵션 추가
export function fetchEntities(params: QueryParams, options?: FetchOptions) {
  // 기존 호출 방식 유지
}

적용 기준:

  • 선택적 파라미터 추가
  • 신규 기능 추가
  • 기존 화면/호출 흐름 유지
  • 기존 데이터 구조와 호환 유지

Major​

기존 사용 코드나 운영 동작을 깨뜨릴 수 있는 변경입니다.

// Major: 필수 파라미터 추가
export function fetchEntities(
  params: QueryParams,
  workspaceId: string,
) {
  // 기존 호출 코드 수정 필요
}

적용 기준:

  • 필수 파라미터 추가
  • API 응답 구조 변경
  • enum/union 값 제거 또는 의미 변경
  • 기본 동작 변경
  • 마이그레이션이 필요한 DB/API 계약 변경

Changesets 기록 방식​

소스코드 변경과 기본 검증이 끝나면, PR을 만들기 전에 변경 범위에 맞는 Changeset을 남깁니다.

Changeset은 major/minor/patch를 판단해주는 도구가 아니라, 개발자가 판단한 semver 의도와 변경 요약을 릴리즈 시스템이 읽을 수 있는 형태로 남기는 기록 파일입니다.

.changeset/example.md
---
"@service/admin": minor
"@service/ui": patch
---

Add equipment status filter to admin dashboard.

- admin: add status filter to equipment list
- ui: fix select component empty-state rendering

작성 기준:

  • 변경된 패키지를 명확히 선택
  • patch/minor/major 판단 근거를 요약
  • breaking change가 있으면 마이그레이션 방법 포함
  • 기능 추가와 리팩터링이 섞이면 PR 또는 Changeset을 분리

패키지 단위 적용 예시​

프로젝트가 monorepo이거나 공용 UI·토큰 패키지를 포함하는 경우, 변경 대상 패키지를 분리해 기록합니다.

packages/
├── web
├── admin
├── ui
└── design-tokens
package.json
{
  "scripts": {
    "changeset": "changeset",
    "version-packages": "changeset version",
    "release": "changeset publish"
  }
}

이 구조에서 개발자는 PR 전에 변경 의도와 패키지 범위를 Changeset으로 기록하고, CI/CD는 릴리즈 시점에 누적된 Changeset을 기준으로 version, changelog, tag, deploy를 처리합니다.

릴리즈 실행 흐름​

Changeset은 PR 전에 작성하지만, 실제 버전 반영과 배포는 릴리즈 시점에 실행됩니다.

공통 흐름은 다음과 같습니다.

노트
  1. 개발자는 코드 변경과 기본 검증을 완료합니다.
  2. PR 생성 전에 변경 범위에 맞는 Changeset을 작성합니다.
  3. PR에는 코드 변경과 Changeset이 함께 포함됩니다.
  4. 리뷰어는 Changeset의 semver 판단 근거를 확인합니다.
  5. 병합된 Changeset은 기준 브랜치에 누적됩니다.
  6. 릴리즈 시점에 CI/CD가 누적 Changeset을 기준으로 version, changelog, tag, deploy를 처리합니다.

즉, 기준 브랜치 병합은 Changeset 누적을 의미하고, 즉시 배포를 의미하지 않습니다. 배포는 프로젝트별 릴리즈 정책에 따라 수동 승인, 정해진 배포 일정, 또는 별도 릴리즈 워크플로우에서 실행됩니다.

AI 활용 범위​

AI는 버전 관리의 초안 작성과 정적 검증에는 활용하지만, 릴리즈 판단을 자동 승인하지는 않습니다.

핵심 기준은 명확합니다.

  • AI는 변경 diff를 보고 검토 후보를 제안할 수 있습니다.
  • AI는 Changeset 누락, 형식 오류, 테스트 실패 같은 정적 신호를 찾을 수 있습니다.
  • AI는 patch/minor/major를 확정하지 않습니다.
  • AI 판단만으로 릴리즈를 진행하거나 차단하지 않습니다.
구분AI 역할최종 판단
Changeset 누락 확인PR 변경 파일 기준으로 기록 필요 여부 제안개발자
Changeset 형식 검증package명, semver 값, markdown 구조 확인개발자
영향 범위 후보 추정API, schema, package export 변경 감지리뷰어
테스트/타입 실패 확인CI 결과를 근거로 릴리즈 위험 신호 표시CI / 리뷰어
patch/minor/major 확정판단 근거 초안만 제시개발자
비즈니스 리스크 판단자동 판단하지 않음개발 리드 / PM
노트

이렇게 경계를 둔 이유는 major/minor 판단이 코드 diff만으로 끝나지 않기 때문입니다. 기존 클라이언트 사용 방식, 운영 정책, 마이그레이션 가능 여부까지 확인해야 하므로 AI는 판단 보조 도구로만 사용했습니다.

결과​

  • PR 단계에서 변경 의도와 영향 범위를 명시해 리뷰 기준을 표준화
  • FMS/BEMS의 기능 변경과 운영 영향 범위를 릴리즈 단위로 추적
  • SAJU:ME monorepo에서 패키지별 변경 범위와 배포 대상을 분리
  • changelog와 tag 생성을 자동화해 릴리즈 이력 추적성 확보
  • AI는 Changeset 누락과 영향 후보 검토에 활용하고, semver 확정 책임은 개발자에게 남겨 품질 기준 유지
