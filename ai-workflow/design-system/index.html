디자인 시스템 운영

디자인이 완전히 확정되지 않은 상태에서 AI Agent를 활용해 UI 자산을 점진적으로 공통화한 전략입니다.

목표는 처음부터 완성형 디자인 시스템을 만드는 것이 아니라, 반복 UI를 식별하고 안정화된 컴포넌트만 공통 레이어 또는 패키지로 승격하는 것입니다.

운영 방식​

초기에는 shared 레이어에서 시작하고, 사용처와 props가 안정화되면 패키지로 승격합니다.

초기
shared/ui/
shared/design-tokens/

안정화 이후
packages/ui/
packages/design-tokens/

적용 기준​

단계판단 기준
UI 인벤토리여러 화면에서 반복되는 UI인지 확인
공통화 후보props 구조와 변형 케이스가 안정적인지 확인
shared 승격특정 도메인 의존성이 제거됐는지 확인
package 승격여러 앱 또는 패키지에서 재사용 가능한지 확인
Storybook 검증독립 렌더링, variant, 접근성 확인

컴포넌트 우선순위​

우선순위대상
v1Button, Input, Select, Textarea, Badge, Card, Modal
v2Tabs, Toast, Dropdown, DatePicker
v3도메인 의존성이 낮은 복합 컴포넌트

공통화 기준은 “많이 보인다”가 아니라 도메인 의존성을 제거해도 의미가 유지되는가입니다.

AI Agent 활용​

AI Agent는 디자인 시스템 전환을 자동 결정하지 않고, 반복 패턴을 찾고 승격 후보를 정리하는 데 사용했습니다.

작업AI 활용
반복 UI 탐색화면별 유사 컴포넌트 후보 수집
props 정리중복 props와 variant 후보 정리
토큰 후보 추출반복 색상, spacing, radius 패턴 수집
리뷰 보조도메인 로직이 공통 컴포넌트에 섞였는지 점검
문서화컴포넌트 사용 기준과 변경 이력 정리

최종 승격 여부는 실제 사용처, 변경 빈도, 디자인 확정 상태를 기준으로 개발자가 판단했습니다.

주석 규칙​

디자인 시스템 후보 파일에는 추적 가능한 주석을 남깁니다.

// [DS] domain:home component:Button ui:button
// [DS] domain:profile component:ProfileModal ui:modal
// [DS] domain:form component:InputField ui:form
필드의미
domain최초 사용 화면 또는 도메인
component컴포넌트 이름
uiUI 유형

이 규칙으로 나중에 인벤토리 재수집, 중복 제거, 패키지 승격 대상을 추적할 수 있습니다.

현재 구조​

shared/
├── design-tokens/
└── ui/
    ├── button/
    ├── input/
    ├── card/
    ├── badge/
    └── select/

packages/
├── design-tokens/
└── ui/
