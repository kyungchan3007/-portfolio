병합 전 리뷰를 표준화한 이유

부족한 인력으로 빠르게 개발하다 보면, 리뷰가 사람의 컨디션과 시간에 따라 들쭉날쭉해지기 쉽습니다. 저는 이 변동성을 없애기 위해 Claude 구현 → Codex 검증·수정 리뷰 절차를 표준화했습니다. 누가 언제 리뷰하든 같은 기준으로, 같은 형식으로 판정되게 만드는 것이 목표였습니다.

기본 절차​

① Claude가 구현 완료
      ↓
② Codex에 검증 요청
      ↓
③ 품질 게이트로 작업 유형 분류
      ↓
④ 검증 기준에 따라 리뷰
      ↓
   ┌──┴──┐
  PASS  HOLD
   ↓      ↓
  병합   Codex가 직접 수정 → 재검증

게이트 매트릭스​

작업 성격에 따라 검토 범위를 먼저 나누고, 그다음 단계별 게이트를 적용하는 패턴을 사용했습니다. 세부 기준을 모두 공개하기보다, 작업 유형에 맞는 검토 축을 고정해 리뷰 편차를 줄이는 데 목적을 뒀습니다.

유형 분류 방식​

분류 패턴설명
기능 변경비즈니스 로직과 상태 흐름 중심으로 검토합니다.
인증·보안 변경접근 제어와 민감정보 처리 중심으로 검토합니다.
UI·표현 변경렌더링 영향 범위와 회귀 여부 중심으로 검토합니다.
운영·배포 변경실행 조건과 운영 안정성 중심으로 검토합니다.

게이트 적용 방식​

G1~G5처럼 단계를 나눈 게이트 패턴을 사용했습니다. 각 단계는 로직, 계약, 보안, 상태 일관성, 회귀 위험처럼 서로 다른 관점을 담당하도록 구성했고, 작업 유형에 따라 필요한 단계만 조합해 검토했습니다.

리뷰 출력 형식​

리뷰 결과가 매번 다른 모양으로 나오지 않도록, 근거와 판단이 한눈에 보이는 고정된 출력 패턴을 사용했습니다. 세부 템플릿을 공개하기보다, 같은 형식으로 기록하고 같은 기준으로 판정하는 흐름을 유지하는 데 중점을 뒀습니다.

재검증 방식​

HOLD가 나온 뒤 수정해 다시 요청하면, 전체를 처음부터 반복하기보다 변경된 범위부터 우선 검증하도록 기준을 잡았습니다. 재검증 시간을 줄이되, 이전 Critical/High 이슈가 남아 있으면 반드시 다시 확인하도록 했습니다.
