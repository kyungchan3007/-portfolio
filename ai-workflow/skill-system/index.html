스킬 시스템

스킬은 AI Agent가 작업 전에 읽는 도메인별 지침 파일입니다.

모든 프로젝트 규칙을 매번 대화로 설명하지 않고, 작업에 필요한 문서만 선택적으로 읽게 만들어 컨텍스트 낭비와 구현 편차를 줄였습니다.

기본 구조​

.agents/skills/{skill-name}/
├── SKILL.md
└── references/
    ├── architecture.md
    ├── api.md
    ├── auth.md
    └── ...
구성역할
SKILL.md도메인 진입점, 읽기 규칙, 공통 원칙
references/작업 유형별 세부 지식
description어떤 요청에서 이 스킬을 사용할지 결정하는 트리거 문장

읽기 원칙​

한 작업에서 모든 문서를 읽지 않습니다.

입력 폼 흐름 변경 → form-flow reference
API 연동 변경     → api reference
인증 흐름 변경    → auth reference
UI 스타일 변경    → ui-style reference

원칙은 SKILL.md + 필요한 reference만 읽는 것입니다. 여러 도메인에 걸치는 작업일 때만 추가 reference를 읽습니다.

구현 전 출력​

구현 Agent는 코드를 작성하기 전에 아래 내용을 먼저 정리합니다.

읽은 문서:
- SKILL.md
- references/api.md

작업 요약:
- 변경할 기능과 영향 범위

레이어 계획:
- model/: 순수 로직
- hooks/: 상태와 데이터 흐름
- ui/: 렌더링 컴포넌트
- entities/: API 타입과 계약

이 단계는 AI가 바로 코드를 쓰기 전에 프로젝트 구조와 영향 범위를 한 번 검토하게 만드는 장치입니다.

운영 중인 스킬​

스킬용도
frontend웹 애플리케이션 구현, 라우트, API 연동, 상태관리
code-review-guard병합 전 diff 리뷰
code-review-workflow작업 유형별 검증 게이트
design-system-hybrid디자인 시스템, 토큰, 공통 컴포넌트
weekly-worklog변경 요약과 주간 작업 기록
prompt-trigger-eval스킬 트리거 오탐/미탐 점검
best-practices성능, 렌더링, 번들 크기 점검
web-design-guidelinesUI 일관성, 접근성, 레이아웃 점검

트리거 품질 관리​

스킬의 description은 라우팅 기준입니다.

너무 넓으면 관련 없는 요청에서도 발동하고, 너무 좁으면 필요한 상황에서 발동하지 않습니다.

description: >-
  웹 애플리케이션 구현 또는 변경 요청에 사용한다.
  라우트, API 연동, 상태관리 요청이 해당된다.
  백엔드 전용 변경, 인프라 전용 작업에는 사용하지 않는다.

prompt-trigger-eval은 should_trigger / should_not_trigger 샘플로 스킬 발동 경계를 점검합니다.

갱신 규칙​

스킬은 고정 문서가 아니라 코드와 함께 바뀌는 운영 문서입니다.

  • 새 도메인이 생기면 reference 추가
  • 반복되는 실수가 나오면 규칙 추가
  • 레이어 구조가 바뀌면 architecture reference 갱신
  • 스킬 오탐/미탐이 발견되면 description 조정
