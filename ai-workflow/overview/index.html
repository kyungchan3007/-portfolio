AI Agent 운영 개요

FMS를 중심으로 적용한 AI Agent 기반 개발·검증 워크플로우입니다.

목표는 AI로 구현 속도만 높이는 것이 아니라, 구현·검증·기록의 책임을 분리해 결과물 품질이 흔들리지 않도록 운영 기준을 고정하는 것입니다.

왜 도입했나​

AI에게 “이 기능 만들어줘”라고만 요청하면 다음 문제가 반복됩니다.

문제영향
관련 없는 파일까지 과도하게 탐색토큰 낭비와 작업 시간 증가
프로젝트 구조를 무시한 구현FSD/VSA 경계 위반, 유지보수 비용 증가
구현한 AI가 스스로 검증자기 코드의 결함을 놓칠 가능성 증가
리뷰 기준이 매번 달라짐병합 판단과 품질 기준이 흔들림
변경 기록 누락주간 작업 내역과 릴리즈 추적성 저하

이를 해결하기 위해 스킬 시스템, 역할 분리, 리뷰 게이트를 하나의 운영 흐름으로 묶었습니다.

적용 범위​

프로젝트적용 내용
FMS기능 구현, 코드 리뷰, 작업 기록까지 AI Agent 워크플로우 적용
SAJU:MEmonorepo 구조, 디자인 시스템, 릴리즈 관리에 스킬 기반 작업 방식 적용
포트폴리오 문서AI Workflow 운영 기준을 문서화하고 재사용 가능한 형태로 정리

전체 흐름​

운영 흐름은 다음 순서로 고정했습니다.

  1. 작업 요청을 도메인별 스킬로 라우팅
  2. 구현 Agent가 필요한 reference만 읽고 코드 작성
  3. typecheck, unit test, e2e, build 등 검증 명령 실행
  4. Reviewer Agent가 변경 diff와 검증 결과를 기준으로 MERGE: PASS/HOLD 판정
  5. PASS 이후 변경 요약과 작업 기록 작성

핵심 구성​

문서역할
에이전트 역할 분리구현, 검증, 기록 Agent의 책임 경계
하네스 + 온톨로지스킬 라우팅과 실행 규칙을 고정하는 구조
스킬 시스템작업 전 필요한 문서만 읽는 방식
디자인 시스템 운영UI 자산을 점진적으로 공통화하는 전략
코드 리뷰 워크플로우작업 유형별 리뷰 게이트와 PASS/HOLD 기준

운영 원칙​

  • 구현과 검증 분리: 구현 Agent는 코드를 작성하고, Reviewer Agent는 diff와 리스크만 검증합니다.
  • 필요한 문서만 읽기: 모든 규칙을 매번 읽지 않고 작업 도메인에 맞는 reference만 사용합니다.
  • 판정 기준 고정: 리뷰 결과는 MERGE: PASS 또는 MERGE: HOLD로 통일합니다.
  • 심각도 통일: 품질 이슈는 Critical / Major / Minor로 분류합니다.
  • 완전 자동 병합 금지: AI 판정은 병합 판단을 보조하며, 최종 병합 책임은 개발자에게 있습니다.

결과​

  • AI 구현 결과를 그대로 병합하지 않고, 독립 리뷰 게이트를 거치도록 표준화
  • 작업 유형별 검증 기준을 고정해 리뷰 품질 편차 감소
  • 필요한 reference만 읽도록 제한해 컨텍스트 낭비 감소
  • 변경 요약과 작업 기록을 분리해 프로젝트 추적성 확보
