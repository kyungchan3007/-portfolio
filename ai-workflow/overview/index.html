왜 AI Agent 워크플로우를 직접 만들었나

이 섹션은 기능 설명서가 아니라, 제가 마감과 인력 사이에서 품질을 지키기 위해 AI 개발 방식을 어떻게 다시 설계했는지에 대한 기록입니다.

상황 — 사람은 부족하고, 마감은 가까웠다​

프로젝트는 인력이 부족한 상태로 굴러가고 있었고, 마감까지 남은 시간은 넉넉하지 않았습니다. 그렇다고 제품 품질을 포기할 수는 없었습니다. 손이 부족하다는 이유로 릴리즈되는 코드의 기준을 낮추고 싶지 않았습니다.

그래서 AI를 개발에 본격적으로 끌어들였습니다. 부족한 손을 AI로 메우면 마감과 품질을 동시에 잡을 수 있을 거라고 생각했습니다.

역설 — AI를 쓰자 오히려 더 느려졌다​

기대와 달리, 아무 장치 없이 AI에게 "이 기능 만들어줘"라고 맡기자 문제가 쏟아졌습니다.

  • 토큰 낭비 — AI가 관련 없는 파일까지 잔뜩 읽으면서 컨텍스트가 계속 부풀었습니다.
  • 엉뚱한 방향 — 프로젝트 규칙을 무시한 코드, 불필요한 추상화가 나왔습니다.
  • 결국 더 오래 걸렸다 — 잘못된 결과물을 되돌리고 다시 설명하는 시간이 쌓여, 실질적으로는 사람이 직접 짜는 것보다 느렸습니다.

AI가 문제가 아니라, AI에게 일을 맡기는 방식이 문제였습니다.

전환 — 하네스 + 온톨로지 엔지니어링​

이 문제를 풀기 위해 하네스(Harness) 와 온톨로지(Ontology) 엔지니어링을 공부했습니다.

  • 하네스 — AI가 매번 다르게 판단하지 않도록 실행 절차(구현→검증→기록)와 판정 형식(MERGE: PASS/HOLD)을 고정했습니다.
  • 온톨로지 — 작업 도메인을 분류하고, 각 작업에 필요한 지식(references/)의 위치를 정의해 "필요한 것만 읽게" 만들었습니다.

곧바로 프로덕션에 적용하지 않고 사이드 프로젝트에서 먼저 실험했습니다. prompt-trigger-eval로 스킬이 엉뚱하게 발동하는지 점검하고, 스모크 테스트로 실제 동작을 확인하고, 그 결과를 보며 지침서를 고치고 스킬 경계를 다시 그었습니다. 이 루프를 작업 완성율이 95% 이상으로 올라갈 때까지 반복했습니다.

자세한 여정은 하네스 + 온톨로지 엔지니어링 문서에 정리했습니다.

결과 — 두 AI를 역할로 나눈 워크플로우​

그렇게 정착한 것이 Claude와 Codex를 역할로 분리한 워크플로우입니다.

AI역할하는 일
Claude구현스킬을 읽고 도메인 규칙을 파악한 뒤 초안 코드 작성
Codex검증 · 수정diff 리뷰, MERGE: PASS/HOLD 판정, 잘못된 코드·구조 직접 수정

같은 AI가 구현과 검증을 겸하면 자기 코드의 맹점을 못 봅니다. 그래서 Claude는 구현만 맡기고, 검증과 잘못된 코드·구조의 수정은 Codex에게 넘겨 독립적인 시각으로 고치게 했습니다.

1️⃣  Claude (구현)      스킬 읽기 → 도메인 규칙 파악 → 초안 코드 작성
2️⃣  Codex (검증·수정)  diff 수신 → 게이트 매트릭스 분류 → PASS / HOLD → 잘못된 부분 수정
3️⃣  Worklog (기록)     변경 요약 → 작업 기록 반영

HOLD가 나오면 Codex가 잘못된 코드·구조를 직접 수정한 뒤 다시 검증해 병합했습니다.

이 섹션에서 다루는 이야기​

  • 에이전트 역할 분리 — 왜 구현과 검증을 다른 AI에게 맡겼나
  • 하네스 + 온톨로지 엔지니어링 — 95% 완성율까지 끌어올린 과정
  • 스킬 시스템 — 지침을 온톨로지 노드로 구조화한 방법
  • 디자인 시스템 운영 — 확정되지 않은 디자인을 점진적으로 시스템화한 전략
  • 코드 리뷰 워크플로우 — 병합 전 검증을 표준화한 방식
  • 자동화 검증 파이프라인 — 검증·리뷰·배포를 이어붙인 최종 구조
