검증·리뷰·배포

AI 작업 방식을 테스트하고, 검증하고, 개선해 본 과정을 정리한 내용입니다. 실제 프로젝트 운영 규칙이나 배포 절차를 그대로 설명하는 문서는 아닙니다.

AI를 개발에 접목했을 때 가장 먼저 마주한 문제는 결과 편차였습니다. 어떤 날은 빠르게 원하는 결과가 나왔지만, 어떤 날은 프로젝트 기준과 맞지 않는 코드가 나오거나 검증이 부족한 결과가 섞였습니다. 그래서 중요한 것은 AI를 쓰는 것 자체보다, 결과를 신뢰할 수 있도록 검증 가능한 흐름을 만드는 일이었습니다.

왜 이 흐름이 필요했는가​

처음에는 AI가 구현한 결과를 사람이 눈으로만 확인해도 충분할 것이라고 생각했습니다. 하지만 실제로는 같은 요청이라도 결과가 달라졌고, 작은 수정처럼 보여도 영향 범위가 넓은 경우가 있었습니다. 이 편차를 줄이지 않으면 속도는 빨라져도 품질은 오히려 불안정해질 수밖에 없었습니다.

그래서 저는 AI를 더 잘 쓰는 방법보다, AI 결과를 어떤 기준으로 확인해야 하는지부터 정리했습니다. 구현과 검증을 분리해 보고, 어떤 변경은 가볍게 확인해도 되는지, 어떤 변경은 더 강한 검증이 필요한지, 결과를 어디까지 신뢰할 수 있는지를 계속 점검했습니다.

무엇을 테스트했는가​

검증 축은 복잡하게 늘리지 않고, 실제로 가장 자주 문제를 걸러낼 수 있는 흐름만 반복해서 확인했습니다.

검증 축확인한 목적
타입 검증계약 불일치와 연결 누락을 빠르게 확인하기 위해 사용했습니다.
단위 테스트핵심 로직이 의도대로 동작하는지 먼저 확인했습니다.
E2E 테스트사용자 흐름이 실제 화면에서 끊기지 않는지 확인했습니다.
스모크 테스트변경 이후 기본 진입과 핵심 동선이 깨지지 않는지 점검했습니다.
빌드 확인최종 결과가 배포 가능한 상태인지 마지막에 확인했습니다.

이 과정에서 중요한 것은 도구 이름보다, 어떤 계층의 문제를 어떤 방식으로 걸러낼 수 있는지 이해하는 것이었습니다.

어떤 흐름을 테스트해봤는가​

실제로는 하나의 방식만 고정한 것이 아니라, 변경 성격에 따라 몇 가지 흐름을 나눠 시험해봤습니다. 아래는 그중 대표적인 예시입니다.

1. 빠른 검증 중심 흐름​

작은 로직 수정이나 영향 범위가 비교적 좁은 변경에서는 가볍고 빠른 검증부터 확인하는 흐름을 사용했습니다.

구현 → 타입 확인 → 단위 테스트 → 리뷰

이 방식은 속도가 중요할 때 유용했고, 계산 로직이나 단일 기능 수정처럼 범위가 좁은 작업을 빠르게 되돌아보는 데 적합했습니다.

2. 사용자 흐름 확인 중심 흐름​

화면 연결, 상태 반영, 사용자 상호작용이 포함되는 변경에서는 실제 동선이 끊기지 않는지 확인하는 흐름을 더 자주 테스트했습니다.

구현 → 단위 테스트 → E2E 확인 → 스모크 테스트 → 리뷰

이 방식은 화면에서 보이는 결과가 중요한 작업에서 효과적이었고, 단순히 코드가 맞는지보다 사용자가 실제로 문제 없이 흐를 수 있는지를 보는 데 초점이 있었습니다.

3. 배포 전 보수적 확인 흐름​

인증, 계약, 상태 동기화, 배포에 가까운 변경에서는 좀 더 보수적인 흐름을 사용했습니다.

구현 → 타입 확인 → 단위 테스트 → E2E 확인 → 빌드 확인 → 최종 리뷰

이 방식은 속도보다 안정성을 우선할 때 사용했고, 영향 범위가 넓은 변경에서 마지막까지 확인해야 하는 항목들을 놓치지 않기 위한 흐름이었습니다.

4단계 검증 프레임​

모든 변경을 같은 무게로 보지 않고, 변경 성격에 따라 아래처럼 단계를 나눠 검증하는 방식을 테스트했습니다.

단계유형기본 검증추가 검증 조건추가 검증
1단계계산/순수 로직 변경관련 unit test 확인계산 결과 재사용, API 매핑 영향영향 화면 unit test 추가 확인
2단계계산 결과 → 렌더링 연결unit test, 화면 smokeCTA 노출, 사용자 흐름, 접근성 리스크e2e smoke, 화면 확인
3단계API/캐시/인증 정책 변경typecheck, unit test, 진입 분기 확인key 변경, redirect, cookie 처리 변경e2e 시나리오 확인
4단계UI/UX만 수정화면 smoke, 접근성 기본 확인상태 변화, 조건부 렌더 포함2단계 또는 3단계로 승격

3단계는 가장 보수적으로 봤고, typecheck 통과를 기본 조건으로 두었습니다.

검증 → 리뷰 흐름​

검증이 통과하면 바로 리뷰 단계로 넘기고, 문제가 있으면 변경 범위 중심으로 다시 확인하는 흐름을 사용했습니다.

단계처리 내용
검증 완료변경 유형 분류, 추가 확인 필요 여부 판단
리뷰 진행변경 성격에 맞는 검토 관점 적용
판정PASS면 진행, HOLD면 수정 요청
재검증변경된 부분 중심으로 다시 확인

어떤 수치로 확인했는가​

이 흐름들은 감각으로만 판단하지 않고, 반복해서 돌려 보면서 대략적인 수치로도 확인했습니다.

확인한 항목대략적인 기준 또는 변화
테스트 커버리지 기준단위 테스트는 80% 이상을 목표로 보고, 부족하면 보강이 필요한 상태로 판단했습니다.
트리거 정확도사이드 프로젝트에서 반복 검증하며 작업 완성율이 95% 이상으로 올라갈 때까지 흐름을 다듬었습니다.
단위 테스트 재검증 시간전체 재실행은 보통 60 ~ 90초 수준이었고, 변경 범위만 다시 확인하면 10 ~ 20초 수준까지 줄일 수 있었습니다.
E2E 재검증 시간전체 실행은 약 120초 수준이었고, 핵심 흐름만 다시 확인하면 20~40초 수준으로 줄었습니다.

이 수치들은 운영 규칙이라기보다, 어떤 방식이 실제로 효율적인지 판단하기 위해 계속 비교해 본 기준에 가까웠습니다.

무엇을 공부하고 검증했는가​

단순히 테스트를 돌리는 것만으로는 부족했습니다. 실제로는 "어떤 변경에 어떤 수준의 검증이 필요한가"를 계속 조정하는 과정이 더 중요했습니다.

  • 순수 로직 변경은 빠른 검증으로도 충분한 경우가 많았습니다.
  • 화면 흐름이나 상태 반영이 엮이면 더 넓은 사용자 흐름 확인이 필요했습니다.
  • 인증, 계약, 캐시, 배포에 가까운 변경은 더 보수적으로 봐야 했습니다.

이 과정에서 단순히 테스트를 추가한 것이 아니라, 검증 범위를 단계적으로 올리거나 줄이는 패턴도 함께 시험했습니다. 작은 수정은 가볍게 확인하고, 영향 범위가 넓어질수록 타입, 테스트, 사용자 흐름, 빌드까지 점진적으로 올려 보는 방식이 더 효율적이라는 판단을 얻었습니다.

이 판단 기준은 한 번에 정해진 것이 아니라, AI가 만든 결과를 여러 번 검토하고 수정하면서 다듬어졌습니다. 어디까지 자동화해도 되는지, 어디서부터는 사람이 더 엄격하게 확인해야 하는지, 어떤 종류의 오류가 반복되는지를 계속 기록하고 조정했습니다.

판정 기준​

리뷰는 심각도 기준으로 간단히 나눠 관리했습니다.

구분의미
Critical런타임 에러, 보안, 테스트 없는 핵심 로직, 계약 위반 등 병합 차단 수준
Major성능 문제, 구조 위반, 중요한 검증 미통과 등 병합 차단 수준
Minor네이밍, 접근성 기본 요구사항, 복잡도 등 경고 수준

최종 판정은 MERGE: PASS 또는 MERGE: HOLD 형태로 정리했습니다.

무엇을 얻었는가​

이 과정을 거치면서 AI를 막연히 빠른 도구로 쓰는 단계에서, 검증 가능한 작업 방식으로 다루는 단계로 넘어갈 수 있었습니다.

  • 재작업이 줄었습니다.
  • 검증 순서를 정리하면서 확인 속도가 빨라졌습니다.
  • 팀이 같은 기준으로 결과를 볼 수 있게 되면서 불안감이 줄었습니다.
  • 좋은 결과가 나온 방식은 이후 실제 프로젝트에 선택적으로 접목할 수 있었습니다.

결국 이 문서에서 중요했던 것은 자동화 자체가 아니라, AI 결과를 신뢰할 수 있게 만드는 기준을 찾고 다듬은 과정이었습니다.
