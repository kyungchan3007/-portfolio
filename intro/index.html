팀을 위한 기준을 만들고 정착시킨 방식​

공통으로 적용한 방식은 단순히 코드를 잘 작성하는 것을 넘어서, 팀 전체가 같은 기준으로 구현하고 검증할 수 있는 구조를 만드는 일이었습니다.

  • 코드 컨벤션과 작업 기준을 먼저 문서화해, 구현 스타일과 리뷰 기준이 사람마다 달라지지 않도록 맞췄습니다.
  • 성능을 체감이 아니라 수치로 확인할 수 있도록 측정 코드를 만들고, 프론트엔드 개발자들이 바로 활용할 수 있게 공유했습니다.
  • 도메인별로 필요한 기술과 스킬을 따로 찾고, 개선 여지가 보이면 사이드 프로젝트에서 먼저 검증한 뒤 결과를 문서화해 팀에 배포했습니다.
  • 문서를 공유할 때는 "이 도메인에 왜 이 스킬이 필요한지", "사용할 때 무엇을 주의해야 하는지", "장점과 단점이 무엇인지", "적용하면 어떤 개선 효과를 기대할 수 있는지"까지 함께 정리했습니다.

AI Agents 워크플로우를 정착시킨 방식​

AI Agents 워크플로우는 처음부터 팀이 자연스럽게 받아들인 방식은 아니었습니다. 사람 대신 AI가 구현과 검증의 많은 부분을 맡는 구조였기 때문에, 팀원들도 "정말 검증이 가능한가"에 대해 보수적으로 볼 수밖에 없었습니다. 저 역시 같은 의문이 있었기 때문에, 바로 현업에 적용하지 않고 먼저 검증 가능한 근거를 만드는 데 집중했습니다.

  • 검색과 리서치를 통해 검증에 필요한 스킬과 지침서를 찾고, AI와의 대화와 직접 검증을 반복하며 기준을 다듬었습니다.
  • Next.js에서 특히 중요하다고 본 검증 포인트인 "서버에만 있어야 할 것이 클라이언트 번들로 새어나가지 않는지"를 중심으로 검증 스킬과 워크플로우를 설계했습니다.
  • 프롬프트 테스트와 스모크 테스트를 반복하며 좋은 결과가 나올 때까지 워크플로우를 계속 개선했습니다.
  • 최종적으로는 검증 기준, 사용 방법, 불안 요소를 줄이는 이유까지 문서화해 팀원들에게 공유했고, 이 과정을 통해 AI 작업 방식에 대한 불안을 줄인 상태에서 도입을 진행했습니다.

BEMS에서 기술 기준을 정착시킨 방식​

프론트엔드 사수가 없는 환경에서 Web Worker와 JSON 기반 동적 계산 방식도 직접 찾아보고 검증해야 했습니다. 블로그, Reddit, 기술 문서를 보며 근거를 모았고, 단순히 "좋아 보이는 기술"이 아니라 BEMS 도메인에 정말 필요한지부터 판단했습니다.

  • Web Worker는 모든 메뉴와 라우터가 지속적으로 데이터를 받아 그리고 있는 BEMS 특성상, 메인 스레드 부하를 줄이기 위한 거의 필수적인 선택이라고 판단했습니다.
  • 그래서 메인 스레드와 백그라운드 스레드가 어떻게 역할을 나누고 결과를 전달하는지 아키텍처를 직접 그리고, 검증한 코드와 측정 결과를 팀원들과 공유했습니다.
  • JSON 기반 동적 계산 방식은 이미 거의 완료된 프로시저 안에 수식과 프로세스를 계속 넣기 어려운 상황에서 선택했습니다. 수식을 JSON 파일에 등록하고 동적으로 계산하도록 바꿔, 수정 범위를 줄이고 정합성 대응을 더 유연하게 만들었습니다.
  • 이 역시 왜 필요한지, 어디에 적합한지, 어떤 장단점이 있는지 문서화해 팀원들이 같은 판단 기준으로 적용할 수 있도록 공유했습니다.

사이드 프로젝트에서 검증한 방식​

PinHouse에서는 SSR 경계, 인증 진입, React Query 캐시 구조 병목을 다듬으며 프론트엔드 기준을 정리했습니다.
SAJU:ME에서는 AI Agents, 운영 도구, Edge 배포, 릴리즈 구조, 디자인 시스템 패키지 구조를 실제 배포·운영 과정에서 먼저 검증했습니다.

이 과정에서 검증된 AI Agents 워크플로우는 FMS에 전체 적용되었고, 배포 전 검증 흐름과 디자인 시스템 패키지 경험은 BEMS, 원격 제어, FMS 전반에 적용되었습니다.
